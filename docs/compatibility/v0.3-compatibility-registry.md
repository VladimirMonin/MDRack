# MDRack v0.3 compatibility registry

- **Status:** Accepted and implemented for the v0.3 compatibility release
- **Baseline:** `e7eeb3cda2d10836f4dffe03c0ec134d448026d5`
- **Applies to:** the v0.3 compatibility release
- **Decision:** [ADR-0002](../decisions/0002-provider-storage-neutral-core.md)

This registry preserves the historical v0.2 importer baseline and defines the current
v0.3 compatibility owners. `SURVIVE` means the named public/import surface remains
usable for one compatibility release and delegates to one canonical core-backed app
path. Removal requires every listed oracle at the exact reviewed revision.

## Global compatibility invariants

- Existing CLI success/error envelopes and `meta.command` remain stable.
- Legacy text/semantic/hybrid DTO key sets, score/rank/nullability, degradation,
  heading arrays, and source locators remain stable where representable.
- `chunk_id` remains a document-unit compatibility alias; it is not repurposed as a
  generic resource ID.
- New resource/unit contracts use logical identities and never expose SQLite record IDs.
- `MDRackEngine` remains an app façade; pure core APIs are separate.
- Wrappers delegate to one implementation. No wrapper retains its own RRF, grouping,
  storage write, ID-generation, query-vector, or compatibility-mapping algorithm.
- Import compatibility is verified from an installed wheel, not only the source tree.

### Safe diagnostic schemas for changed surfaces

These schemas are implemented v0.3 contracts. Their S8 implementation updated
`docs/cli-contracts.md`; S11R1 freezes the complete installed-package compatibility
inventory below without changing those schemas.

- Eval success data is
  `{query_set:{kind:"file",query_count},k,results,summary}`. Each result contains
  `{case_ordinal,mode,k,recall_at_k,mrr,precision_at_k,ndcg_at_k,retrieved_count,
  expected_count,conditions_met,status,reason_code?}`. `query_set` identifies the
  input class, never its path or basename. `case_ordinal` is a one-based integer;
  user-controlled query IDs are not emitted. Errors keep the outer envelope and a
  stable `error.code`; `error.message` is a fixed public message selected by code.
- Status data is
  `{generation_state,files_count,chunks_count,embeddings_count,active_profile,
  profile_model,profile_dimensions,configured_model,configured_dimensions,
  endpoint_configured,endpoint_profile_recorded,endpoint_match,schema_version}`.
  Endpoint booleans are nullable when comparison is unavailable. No endpoint family,
  host, port, URL, or path is emitted.
- Doctor keeps `{ok,summary,findings}`. Every finding is
  `{severity,code,message,details}` where `message` is fixed by `code` and `details`
  is an allowlisted map of booleans, counts, dimensions, profile/model identifiers,
  fingerprints, migration versions, generation state, and stable reason codes.
  Keys containing `path`, `endpoint`, `url`, `host`, `port`, `query`, `content`,
  `vector`, `body`, `exception`, `expected_*` private values, or `actual_*` private
  values are forbidden.
- Model/status diagnostics retain documented model keys and instance IDs, but all
  failures use `{code,message,details?}` with a fixed message and allowlisted stable
  reason/status/capability values. Provider response objects are mapped field by
  field; they are never serialized recursively as a diagnostic shortcut.

## Public behavior oracles

The compatibility owner must maintain these oracles through S6 and all later stages:

1. byte-equivalent CLI/engine results for text, semantic, and hybrid compatibility paths;
2. exact JSON envelope and command-name tests for success and error families;
3. exact result key set, aliases, score/rank/nullability, stable ordering, degradation,
   `heading_path` array, and complete `SourceLocator` tests;
4. installed-wheel imports for registry surfaces marked `SURVIVE`;
5. scan/rebuild result-field parity and no source mutation;
6. no SQLite record IDs in new resource/unit DTOs;
7. one canonical production owner for fusion, grouping, ID generation, query-vector
   preparation, compatibility mapping, and persistence.
8. exact changed-surface schemas above, recursive forbidden-key scans, and distinct
   sentinels for query/content/path/endpoint/vector/metadata/provider/exception data;
9. source importer inventory for eval reports, status, doctor, model/status, and
   support/release JSON, plus matching `docs/cli-contracts.md` updates in S8/S11.

## Markdown-specific replacement oracle

Compatibility of the legacy Markdown modules does not preserve the v0.2 asset
product behavior. The final production path must prove that alt/textual alias is
projected once as prose; paths, titles, dimensions, targets, and numeric aliases are
absent; referenced files are never touched; surrounding text is not duplicated;
source bytes and IDs are stable; and no production `Asset`, `AssetReference`,
`AssetGraph`, `IMAGE_REFERENCE`, `image_reference`, asset storage call, or image
resource remains from Markdown scan.

## Evidence and non-claims

Source-tree imports prove only source compatibility. Installed-wheel checks are
required for package compatibility. Fake providers prove orchestration, not live LM
Studio behavior. The checked-out v0.3 implementation includes the compatibility
mapper, core API, resource schema, and direct image command. No compatibility surface
is claimed removed; the deprecation interval remains in force.
