# v0.4 W5-CI contract

Status: extracted contract for the W5-CI implementation wave.

This document is the source contract for the offline release and verification
workflow. It reconciles the v0.4 roadmap, the current architecture and quality
instructions, the standalone distribution and resource UX changes, the privacy
oracle, and the W5 offline release evidence. It does not authorize live
provider calls, remote execution, destructive cleanup, or changes to existing
live jobs.

## Scope and non-goals

W5 proves reproducible offline repository and packaging behavior. It does not
prove Windows or Python 3.12 behavior until that exact cell has run on that
host. It does not prove LM Studio, OCR, Whisper, VLM, external services, a real
vault, or visual/acoustic quality. Provider-backed evaluation remains an
explicitly separate live/network/cost gate.

The default path is no-live:

- `UV_OFFLINE=1` is set for the workflow and subprocesses.
- Dependency installation uses the lockfile with `uv sync --all-extras --frozen --offline`.
- Package installation uses `uv pip --offline` and only the produced artifact directory.
- No provider, HTTP endpoint, source corpus, or remote execution is contacted.
- A live evaluator must not run by omission or by a default script path. Its
  explicit confirmation guard must return a non-zero status before importing or
  contacting the provider.
- Any network attempt, live/provider call, or fallback to an online index is a
  hard failure, not a warning.

## Supported distribution/install cells

These are the four supported distribution cells. Each cell must produce and
independently audit a wheel and an sdist (eight artifacts total):

| Cell | Distribution | Offline install contract |
|---|---|---|
| application | `mdrack` | Installs the three local distributions and exposes the CLI and Python API. |
| reusable core | `mdrack-core` | Installs independently and exposes only provider/storage-neutral core contracts. |
| media records | `mdrack-media` | Installs with `mdrack-core` and exposes provider-free records and builders. |
| SQLite adapter | `mdrack-sqlite` | Installs with `mdrack-core` and exposes the standalone catalog/search adapter. |

The package audit must fail if an artifact has the wrong declared name, version,
local dependency set, missing wheel/sdist pair, or if a standalone artifact
contains the root `mdrack/` package. Installed smoke must clear `PYTHONPATH`,
run outside the repository import path, verify distribution version and module
location, and fail on any import or command error.

## Execution matrix and evidence policy

The declared workflow execution matrix is:

| Host | Python | Status rule |
|---|---:|---|
| Linux (`ubuntu-latest`) | 3.11 | Evidence only when this cell actually runs and passes. |
| Linux (`ubuntu-latest`) | 3.12 | Evidence only when this cell actually runs and passes. |
| Windows (`windows-latest`) | 3.11 | Evidence only when this cell actually runs and passes. |
| Windows (`windows-latest`) | 3.12 | Evidence only when this cell actually runs and passes. |

The matrix is a declaration of coverage, not a claim that all cells have run.
A report must list every cell as `passed`, `failed`, `not_run`, or `blocked` and
must not promote local Linux evidence to Windows or Python 3.12 evidence.
`fail-fast: false` is required so all selected cells report independently; a
failed cell still makes the workflow fail.

Local Linux reproduction uses the same commands and the same offline flags as
CI. Windows commands must be executable in the selected shell (the current
workflow uses Bash-compatible commands) and must not depend on POSIX-only path
or process behavior outside that shell.

## Strict gate contract

All gates are fail-closed: a non-zero exit, missing required artifact, test
collection error, warning promoted by the tool, privacy sentinel hit, network
attempt, or documentation contract mismatch fails the cell. Gates run in this
order after frozen offline environment installation:

1. **Strict source lint**

   ```bash
   uv run ruff check src/ tests/ packages/mdrack-core/src/ packages/mdrack-media/src/ packages/mdrack-sqlite/src/
   ```

   Any Ruff diagnostic is failure. Do not auto-fix in CI and do not downgrade
   the configured E/F/W/I rules.

2. **Strict core/adapter typing**

   ```bash
   uv run mypy packages/mdrack-core/src/mdrack_core packages/mdrack-sqlite/src/mdrack_sqlite
   ```

   Any mypy error is failure. This gate is scoped to the standalone typed
   packages; it must not be reported as whole-repository typing evidence.

3. **Unit and integration lane, excluding separately classified lanes**

   ```bash
   uv run pytest -m 'not e2e and not privacy'
   ```

   Collection errors, failures, xfail misuse, or unexpected deselection are
   failures. This lane includes ordinary unit/integration/packaging tests and
   remains offline.

4. **Offline E2E lane**

   ```bash
   uv run pytest -m e2e
   ```

   E2E means local components, fixtures, and fakes only. It must exercise public
   behavior and package/source boundaries where covered; it must never be
   described as live or real-source evidence. A provider/network call is a
   failure even if the assertion otherwise passes.

5. **Privacy lane**

   ```bash
   uv run pytest -m privacy
   ```

   The lane must fail on supplied query/content/path/root/endpoint/vector,
   metadata/facet, exception, or other registered sentinel leakage across
   success, empty, degradation, failure, and cleanup outputs. Do not print raw
   failing payloads into reports. Privacy tests are explicit and are not hidden
   inside the ordinary lane.

6. **Dependency and architecture boundaries**

   ```bash
   uv run python scripts/check_no_forbidden_deps.py
   uv run python scripts/check_core_boundaries.py
   uv run python scripts/check_sqlite_boundaries.py
   uv run python scripts/check_media_boundaries.py
   uv run python -m compileall -q scripts src packages/mdrack-core/src packages/mdrack-media/src packages/mdrack-sqlite/src
   ```

   Any forbidden import, reverse package edge, media/provider boundary breach,
   or compilation error is failure. These checks do not authorize changing the
   architecture to make a gate green.

7. **Offline build and installed smoke**

   ```bash
   uv run python scripts/offline_release_matrix.py \
     --output-dir "${TMPDIR:-/tmp}/mdrack-release-artifacts" \
     --smoke \
     --expected-manifest docs/evidence/w5-offline-release-matrix.json
   ```

   The script must build all four distributions as wheel and sdist, audit
   metadata and package isolation, run isolated smoke cells, and record zero
   network attempts. A build error, hash mismatch, install error, source-tree
   import, missing artifact, or non-zero smoke command is failure. The generated
   output directory must remain outside the source checkout; it is disposable
   evidence and is not a source deliverable.

8. **Documentation and whitespace**

   ```bash
   test -s docs/evidence/w5-offline-release-matrix.md
   test -s docs/evidence/w5-offline-release-matrix.json
   uv run python scripts/check_release_docs.py
   git diff --check
   ```

   Missing/empty evidence, invalid privacy-safe manifest, missing required
   terminology, or whitespace errors are failure. Documentation must preserve
   explicit non-claims for unexecuted matrix cells and stronger evidence
   boundaries.

For a full local repository acceptance run, also run the project baseline gates
from `AGENTS.md` (`uv run pytest`, the two source Ruff commands, core/sqlite
mypy, forbidden-dependency and boundary checks, and `git diff --check`). The
W5 lanes are the reproducible CI contract; a partial local run must identify
which commands were skipped and why.

## Markers and lane rules

`e2e` and `privacy` are registered pytest markers. Tests under `tests/e2e/`
are collected into the E2E lane by `tests/e2e/conftest.py`; privacy contract
modules set `pytestmark = pytest.mark.privacy`. Tests without either marker
belong to the ordinary lane. Marker selection is explicit and deterministic;
no default pytest configuration may select live tests.

The existing `scripts/verify.sh` and `scripts/verify.ps1` remain offline
verification helpers and must not invoke `scripts/live_lmstudio_eval.py`.
The live evaluator remains opt-in and confirmation-guarded, outside W5-CI.

## File-scope restrictions

This extraction task may add or edit only this contract artifact. Subsequent
W5 implementation cards may touch only their explicitly assigned paths, with
one writer per shared path:

- workflow: `.github/workflows/offline-release-matrix.yml`; add/refine the
  offline matrix only; do not edit or replace existing live jobs;
- configuration/markers: `pyproject.toml`, `tests/e2e/conftest.py`, and the
  specifically assigned privacy/unit test files;
- offline tooling: `scripts/offline_release_matrix.py` and
  `scripts/check_release_docs.py`; no provider calls and no new dependencies;
- evidence/docs: `docs/evidence/w5-offline-release-matrix.md`,
  `docs/evidence/w5-offline-release-matrix.json`, and W5 contract documents;
- tests: `tests/unit/test_offline_release_matrix.py`,
  `tests/unit/test_offline_scripts.py`, plus specifically assigned privacy and
  E2E regression files.

The following are read-only for W5-CI unless a separate scoped card explicitly
opens them: production `src/` and `packages/` implementation code, SQL
migrations and databases, `scripts/live_lmstudio_eval.py`, source corpora and
user data, credentials/`.env`, lockfile, existing live workflows, and Git
history. Do not add GUI/web/MCP/daemon code, remote runners, provider clients,
new persistent stores, binary media, or package-manager workflows. Do not
commit generated `.release-artifacts/`, caches, logs, databases, or private
paths.

## Required report language

A completed report must separate:

- commands and cells actually executed;
- offline/local-components/installed-package evidence boundaries;
- `not_run` Windows or Python 3.12 cells;
- explicit non-claims about live providers, real source, visual/acoustic quality,
  and external runtimes;
- failures with the exact gate and command, without leaking private inputs.
