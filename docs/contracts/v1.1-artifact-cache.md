# MDRack 1.1 artifact cache contract

Status: application-layer contract for the offline/local 1.1 cache slice.

## Boundary

The artifact cache is a standalone filesystem derivative owned by the application layer. It is not part of `mdrack-core`, the SQLite catalog, a source file, or a host database. Cache hit, miss, corruption, verification, and deletion never change retrieval-catalog semantics.

The cache can hold normalized transcript artifacts, passages, frame manifests/captions, vectors, and other immutable bytes. Payload interpretation remains the producer's responsibility.

## Complete identity

Every `ArtifactCacheKey` contains:

- `artifact_kind`;
- source-content fingerprint;
- producer fingerprint;
- model fingerprint;
- prompt fingerprint;
- configuration fingerprint;
- preprocessing fingerprint.

Every fingerprint is an opaque lowercase `sha256:<64 hex>` value. Raw source, prompt, model configuration, paths, provider bodies, and credentials are rejected as key components. The entry address is SHA-256 over canonical JSON of the complete key. A change to any component produces a miss.

`OptionalTextProvider` additionally verifies that the supplied prompt fingerprint matches the actual prompt. It derives model identity from provider/model/endpoint family and folds `max_tokens` plus the output bound into the runtime configuration fingerprint.

## Storage and immutability

Entries use two-level content-addressed directories:

```text
<cache-root>/<digest-prefix>/<key-digest>/
  metadata.json
  payload.bin
```

Directories are owner-only and files are created with mode `0600`. Publication writes and fsyncs a private temporary directory, then renames it atomically. A valid existing entry is never overwritten, including when a later producer returns different bytes for the same key. Root, shard, entry, metadata, payload, scan, and discard operations are anchored to opened directory descriptors; shard and entry components are opened with no-follow semantics. A symlinked or non-directory shard is corrupt and is never traversed for reads, writes, verification, or recovery.

Metadata contains only the cache schema, opaque key fingerprints, key digest, payload digest, and payload byte count. Payload bytes may be private and must not be logged or returned by diagnostics.

## Lookup and corruption

- missing identity: `miss`;
- valid identity and payload digest: `hit`;
- malformed metadata, key mismatch, missing payload, size mismatch, symlink, or digest mismatch: `corrupt`.

Normal lookup discards only the corrupt cache entry and returns `corrupt`, allowing the caller to rebuild it. `verify()` is read-only: it reports corruption but does not delete or repair anything. Provider payload envelopes also undergo schema/type validation; a physically valid but semantically invalid provider artifact is discarded and regenerated.

A durable provider hit suppresses transport execution even across provider instances. Partial caches request only missing or corrupt identities.

## Configuration

```toml
[cache]
enabled = true
directory = ".mdrack/artifacts"
max_entry_bytes = 104857600
```

Relative directories resolve against the CLI `--root`. Durable provider caching is explicit dependency injection through `artifact_cache` and `cache_context`; constructing/importing a provider performs no filesystem or network work.

## CLI contract

- `mdrack cache status` returns only enabled state, entry count, and payload byte count.
- `mdrack cache verify` hashes entries and returns count-only validity data; corruption exits non-zero and remains untouched.
- `mdrack cache purge --confirm` is the only destructive whole-cache operation. The configured root must not equal or contain the selected project/store/catalog root. Purge requires the exact cache marker, rejects foreign or unsafe content, and removes only validated cache-owned shard structure while retaining the dedicated root and marker.
- `mdrack cache purge` without `--confirm` fails without mutation.

No automatic eviction or cleanup occurs. Confirmed purge is exercised only in disposable test fixtures; no real project cache is deleted by the A1 offline evidence run.

## Evidence boundary

The A1 tests use temporary local filesystems and injected transports. They prove deterministic offline behavior, local cache persistence, call suppression, drift/miss handling, corruption recovery, no-follow shard containment, protected-root rejection, cache-owned purge in disposable fixtures, source/catalog byte invariance, and privacy-safe CLI output. They do not claim live providers, paid/network behavior, private corpus behavior, Windows, Python 3.12, installed-package behavior, or destructive cleanup of a real cache.
