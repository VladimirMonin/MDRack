# MDRack 1.2 unified text search contract

## Status and scope

This is the current public contract for MDRack 1.2.0 unified text retrieval. It
covers one ready resource-core generation containing only derived text for
Markdown notes, audio transcripts, video transcripts, video-frame captions, and
explicit image caption/OCR text. It is additive to the v1.1 surfaces.

```text
mdrack search QUERY --scope all|notes|audio|video|frames|images
mdrack find-similar RESOURCE_ID --scope all|notes|audio|video|images
```

The embedded equivalents are:

```python
await MDRackEngine.search_unified(query, scope="all", mode="hybrid", limit=20)
MDRackEngine.find_similar_resource(resource_id, scope="all", limit=20)
```

## Unified search

`search --scope` selects the unified text contract only when present. It accepts
`all`, `notes`, `audio`, `video`, `frames`, or `images` and returns resource-level
results. `all` is a text-only union; it does not compare image pixels, audio
waves, video frames as images, or any raw-media feature.

`--mode text|semantic|hybrid` remains available:

- `text` uses the ready generation's lexical branch;
- `semantic` queries the configured embedding provider and returns an empty,
  degraded result when a compatible query vector/space is unavailable;
- `hybrid` uses lexical and semantic branches with the existing RRF owner, or
  retains lexical results with a stable degradation reason when its semantic
  branch is unavailable.

For all three modes, result order is deterministic for the same ready generation,
query, scope, and persisted vectors. The result data contains `query`, `mode`,
`scope`, `target="resource"`, `results`, `total_count`, `degraded`, and
`degraded_reason`. Each result contains `resource_id`, `resource_kind`, `score`,
`rank`, and one or more `evidence` records. Evidence contains only logical IDs,
unit kind, branch ID, and a portable locator. It does not disclose SQLite record
IDs, absolute local paths, source text, captions/OCR text, vectors, provider
payloads, or raw exceptions.

Scope behavior is exact:

| Scope | Resource/unit selection |
|---|---|
| `notes` | Markdown document text |
| `audio` | Audio transcript time segments |
| `video` | Video transcript time segments, excluding frame captions |
| `frames` | Video-frame caption text only |
| `images` | Explicit image textual whole-resource units only |
| `all` | All listed text representations |

`--scope` cannot be combined with the resource-preset, target, metadata-filter,
or low-level resource scope options. Invalid combinations return the fixed
`VALIDATION_ERROR` envelope with `Unified search options are invalid`.

## Provider-free resource similarity

`find-similar` resolves exactly one stored textual whole-resource vector from the
specified logical resource ID, filters candidates by scope before the final limit,
and never calls an embedding provider. The `frames` alias is deliberately invalid
for this command because frame captions are query-search evidence rather than a
resource-level whole-resource similarity identity.

The result data contains `query_resource_id`, `scope`, `results`, `total_count`,
`degraded`, and `degraded_reason`; each result uses the same privacy-safe resource
and evidence shape as unified search. Missing or ambiguous stored textual identity,
incompatible vector space, unavailable resource, or adapter failure produces a
stable empty degraded result rather than a fabricated match. `find-similar` only
uses textual whole-resource vectors; visual and acoustic vectors are excluded.

Invalid scope values return the fixed `VALIDATION_ERROR` envelope with `Unified
similarity options are invalid`.

## Rebuild and compatibility

A 1.2-complete store is created by rebuilding a fresh active generation through
the normal indexing and ingestion workflows. Existing rows are not backfilled or
mutated ad hoc, and the 1.2 capability introduces no SQL migration. Rebuilding a
source copy must preserve source bytes while replacing only derived index data.

The legacy v1.1 `search`, `search --catalog`, `image search`, `similar`,
`resource similar`, and `resources similar` commands retain their documented
contracts. The unified search path is additive and does not alter legacy result
shapes.

## Evidence boundary

The release evidence for this contract is offline/local: deterministic fake
vectors, local SQLite/filesystem components, and an installed-package smoke prove
wiring and packaging for the exercised Linux environment. It does not claim live
LM Studio semantic quality, raw-media recognition, acoustic search, pixel
similarity, Windows execution, hosted publication, or a private real-source
corpus run. A private real-corpus smoke requires separate explicit data
authorization and must use copies with source-hash invariants.
