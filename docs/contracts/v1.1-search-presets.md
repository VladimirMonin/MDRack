# MDRack 1.1 search presets and textual similarity

## Status

This contract defines application-owned orchestration over the existing core search,
branch grouping, weighted RRF, and whole-resource similarity owners. It does not add
another ranking or similarity algorithm.

## Explicit resource-search presets

The caller selects exactly one preset. MDRack does not classify the query or switch
presets automatically.

| Preset | Transcript signal | Frame-caption signal | Metadata lexical signal |
|---|---:|---:|---:|
| `speech_first` | 1.0 | 0.35 | 0.15 |
| `balanced` | 1.0 | 1.0 | 0.20 |
| `frames_first` | 0.6 | 1.0 | 0.15 |

Transcript and frame-caption signals split into lexical `0.4` and semantic `0.6`
fractions. Metadata has no semantic branch. The exact hybrid branch weights are:

| Branch | `speech_first` | `balanced` | `frames_first` |
|---|---:|---:|---:|
| `transcript_text` | 0.40 | 0.40 | 0.24 |
| `transcript_semantic` | 0.60 | 0.60 | 0.36 |
| `frame_caption_text` | 0.14 | 0.40 | 0.40 |
| `frame_caption_semantic` | 0.21 | 0.60 | 0.60 |
| `metadata_text` | 0.15 | 0.20 | 0.15 |

Every branch uses `max(100, result_limit * 10)` candidates. Branch scopes are
fixed to timed transcript passages, frame captions, or metadata text. The global
resource/media/namespace/facet scope is intersected by core before candidate limits.
Core groups candidates by resource inside each branch before weighted RRF, so one
resource contributes at most once per branch even when it owns many passages or
frames. Result evidence remains bounded by the core evidence limit.

Public surfaces:

- `MDRackEngine.search_resource_content(...)`;
- `mdrack search ... --catalog CATALOG --target resource --preset PRESET`.

`MDRackEngine` accepts the same typed `MetadataFilters` used by other retrieval
surfaces. CLI `--tag`, `--meta`, `--meta-any`, and `--meta-none` values are parsed
and compiled into the preset base scope before any branch candidate limit. Invalid
paths or scalar values retain the fixed payload-free metadata validation error.

Semantic and hybrid modes use the caller-selected text embedding provider and an
unambiguous matching space fingerprint. Semantic failure is explicit. Hybrid may
retain lexical results with a degradation reason. Empty or non-finite provider
vectors, provider exceptions, missing or ambiguous spaces, resolver failures,
adapter errors, and adapter timeouts are normalized to fixed payload-free
degradation reasons; raw provider, resolver, and adapter exceptions do not escape.

## Whole-resource textual similarity

The explicit basis is always `textual_content`. The query identifies an existing
`whole_resource` text unit and its vector space. The caller must also identify the
stored aggregation contract:

- `direct_text_v1` for a short normalized text embedded directly;
- `token_weighted_centroid_v1` for a normalized token-weighted centroid of passage
  vectors.

MDRack validates all of the following before similarity search:

1. the query unit exists and is a text `whole_resource` unit;
2. its stored source basis is a recognized textual projection;
3. an explicitly stored recognized aggregation is present and exactly matches the
   requested contract;
4. the vector exists in the requested space;
5. the requested fingerprint and vector dimensions resolve to that exact space.

A missing or mismatched identity returns a stable degraded result and no candidates.
Candidate scope is intersected with text modality before adapter limits, and
candidate units must also carry a recognized textual basis and one of the two
recognized persisted aggregation identities. Higher-scoring visual, acoustic, or
unidentified whole-resource vectors therefore cannot crowd out textual candidates.
The app then delegates the actual vector scoring, same-resource exclusion,
ordering, and safe degradation to the core `ResourceDiscoveryService`.

Current transcript and combined-video ingestion persist a whole-transcript unit.
Short normalized text embedded directly is marked `direct_text_v1`; transcripts
over the bounded direct-text policy are built from the existing normalized
token-weighted centroid and marked `token_weighted_centroid_v1`. The marker is
persisted on the whole unit and its representation and participates in logical
identity; callers cannot relabel an existing vector.

The result contains `query_resource_id`, `query_unit_id`, `space_id`,
`similarity_basis="textual_content"`, the explicit `aggregation`, deterministic ranks,
and bounded evidence with branch, unit, representation, and portable locator fields.
Exact byte duplicates remain a separate operation.

Public surfaces:

- `MDRackEngine.find_textually_similar_resources(...)`;
- `mdrack resource similar QUERY_UNIT --catalog CATALOG --space-id SPACE \
  --embedding-fingerprint FINGERPRINT --aggregation direct-text|token-weighted-centroid \
  --basis textual-content`.

The operation compares text projections only. It makes no claim about pixels, scenes,
sound, voices, music, or other non-text source properties.

## Boundaries and non-claims

- No provider call is made by whole-resource similarity; it reuses a stored vector.
- Preset semantic branches may call only the configured text embedding provider.
- No storage schema, migration, vector database, reranker, query classifier, or
  provider dependency is added to core.
- No live provider, private corpus, Windows, or Python 3.12 result is claimed by the
  offline tests for this contract.
