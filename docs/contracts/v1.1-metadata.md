# MDRack 1.1 metadata normalization, projection, and exact-filter contract

Status: M2 implementation contract

## Scope

Markdown YAML frontmatter is normalized before any storage mutation. Supported
values retain JSON distinctions between strings, integers, finite floats,
booleans, nulls, arrays, and objects. YAML dates and datetimes become ISO
strings; tuples preserve order; sets use canonical-JSON ordering.

The normalizer is `metadata-json-v1`. Its default bounds are:

| Limit | Default |
|---|---:|
| Canonical serialized source metadata | 65,536 UTF-8 bytes |
| Nesting depth | 8 |
| Keys per object | 1,000 |
| Items per array/set | 1,000 |
| String or key | 16,384 UTF-8 bytes |

Unknown objects and binary values are never stringified. Non-finite floats,
unsupported values, invalid keys, and bounded fields are omitted under
`warn_and_continue`. If the final source object exceeds the serialized limit,
the whole source object is omitted. `fail_resource` raises a value-free
`MetadataNormalizationError` instead.

Diagnostics contain only stable categories and per-category aggregate counts. They do not
contain metadata keys, values, YAML exception text, paths, or source excerpts.
Canonical JSON uses sorted keys, compact separators, UTF-8, and finite numbers;
its SHA-256 fingerprint and the normalization-policy fingerprint are stable for
the same input and policy.

## Markdown behavior

A closed frontmatter fence is excluded from body parsing even when YAML is
malformed. Under the default policy the body still indexes, source metadata is
empty, and `METADATA_PARSE_FAILED` is carried as a safe diagnostic. Duplicate
YAML keys are treated the same way. Explicit strict parsing rejects the
resource.

The normalized data travels through:

```text
MarkdownItParser → Document.frontmatter → PreparedFile.source_metadata
→ ResourceRecord.metadata.source
```

The legacy `PreparedFile` path also normalizes and carries its existing
string-valued frontmatter. Legacy parsing is retained only for compatibility;
full nested YAML fidelity belongs to the default markdown-it path.

## Resource envelope

`ResourceRecord.metadata` adds:

- `source`: normalized user metadata only;
- `ingestion`: adapter, parser, chunker, normalizer and safe fingerprints;
- `derived`: key count, total diagnostic count, ordered diagnostic categories,
  and a deterministic category-to-count mapping only. The total equals the sum
  of the category counts; repeated failures in one category are not collapsed.

Frozen v0.3 parser/chunker/path keys remain at the top level for compatibility.
No user values are copied into those keys or into representation/search-unit
metadata.

`ResourceRecord.title` remains canonical and separate. Source metadata does not
enter body chunks, snippets, primary representation text, chunk embedding text,
or vectors. A metadata-only edit changes the source and metadata fingerprints
but not body-derived chunk identities or retrieval/embedding inputs.

## Projection policy

Projection is a pure app-owned operation over normalized `source` metadata. A
policy is an ordered set of unique RFC 6901 JSON Pointers. `~0` selects `~`,
`~1` selects `/`, and array indices use canonical decimal form. Missing paths or
source values incompatible with a projection mode produce no derived value; raw
source metadata remains stored unchanged.

Supported modes are `store_only`, `canonical_title`, `facet`, `facet_many`,
`lexical_text`, and `ignore`. `facet` accepts one scalar. `facet_many` accepts
only an array containing scalars; arrays containing objects are not partially
flattened. Duplicate projected facets are removed deterministically. The default
policy maps `/title` to the canonical title, `/tags` to `tag` facets, and
`/aliases` to lexical values reserved for the M3 representation slice. Unknown
fields remain store-only.

Every policy has a canonical SHA-256 fingerprint. The fingerprint is persisted
in `ResourceRecord.metadata.ingestion.projection_policy_fingerprint` and is also
the producer fingerprint of each source-origin facet.

## Typed scalar facet codec

Existing core facets remain `(namespace, string value)` records. The app-owned
`typed-scalar-v1` codec preserves exact JSON scalar identity without a schema
change:

| JSON type | Stored prefix | Example |
|---|---|---|
| string | `s:` plus canonical UTF-8 percent encoding | `"3"` → `s:3` |
| integer | `i:` plus canonical decimal | `3` → `i:3` |
| float | `f:` plus canonical JSON float | `3.0` → `f:3.0` |
| boolean | `b:true` / `b:false` | `false` → `b:false` |
| null | `z:null` | `null` → `z:null` |

Non-finite floats, objects, and arrays are not scalar facet values. Display uses
canonical JSON scalar text and parses through the same codec, so ingest, query,
and display round-trip without conflating `"3"`, `3`, and `3.0`.

## Exact filters

App `MetadataFilter` values compile through the same typed-scalar codec into the
existing core `SearchScope.facets_any`, `facets_all`, and `facets_none` fields.
The Memory and SQLite adapters therefore apply the filters before each branch's
candidate limit. SQLite uses indexed `EXISTS`/`NOT EXISTS` facet predicates; it
does not scan `metadata_json`, call `json_extract`, or post-filter top-k results.
No range, regex, JSONPath, sorting, or metadata group-by semantics are exposed.

Projection configuration is validated under `[metadata]` with ordered
`[[metadata.projections]]` entries. Facet modes require a namespace; other modes
forbid it; duplicate paths are rejected.

## Boundaries and non-claims

M2 stores source facets and compiles exact filters. Metadata lexical
representations, public inspection/CLI commands, and full policy-driven rebuild
selection remain M3 or later work. This contract adds no schema migration,
JSON post-filter, network or provider call, source write-back, private-corpus
claim, Windows claim, or Python 3.12 claim.
