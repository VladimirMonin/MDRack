# ADR-0011: Freeze MDRack 1.1 application contracts on the current baseline

Status: Accepted for staged implementation
Date: 2026-07-20

## Context

The proposed 1.1 plan describes a delta from a completed 1.0 and uses the
illustrative source root `app/src/mdrack`. The checked-out repository does not
have that shape or a final 1.0 release:

- the application owner is `src/mdrack`;
- `mdrack-core`, `mdrack-media`, and `mdrack-sqlite` are implemented standalone
  packages whose published contract identities remain `1.0.0-rc.1`;
- the application package remains `0.3.0`;
- the v0.4 offline packet explicitly records final 1.0 as not ready;
- current core, media, SQLite, prepared-resource import, resource search,
  textual similarity, and provider-neutral boundaries already cover much of the
  earlier roadmap.

Treating the plan's assumed 1.0 as completed would create false evidence.
Replaying the existing foundations would create duplicate owners and risk
breaking accepted contracts.

## Decision

1. Revision `989aaaeaf029d7bdd2d9704ae578c9a19efe0f94` is the B0 source baseline.
   It is a development baseline for 1.1, not a 1.0 release or tag.
2. The only implementation root for the standalone application is
   `src/mdrack`. The illustrative `app/src/mdrack` path is not a repository
   target.
3. The existing `mdrack_core`, `mdrack_media`, and `mdrack_sqlite` public
   contracts and algorithms are foundations to preserve. Application work may
   compose them but may not fork or silently widen them.
4. SQLite remains the only persistence implementation. The 1.1 application
   stages must first use existing metadata JSON, facets, scoped search, complete
   replacement, and clean-catalog capabilities without a migration.
5. Missing 1.1 behavior is application-owned and is frozen by
   [the application contract](../contracts/v1.1-application-contract.md):
   metadata normalization/transport, metadata projection and exact-filter
   compilation, allowlisted metadata UX, transcript readers and orchestration,
   complete video composition, search presets, artifact cache, and compatible
   manifest export/public-surface consolidation.
6. Existing entry points remain `MDRackEngine`, `PreparedResourceCatalog`, and
   the Click CLI. No second application facade named `MDRack`, daemon, server,
   provider method in core, or source-specific media dependency is introduced
   by this freeze.
7. Each downstream stage owns its implementation and tests. This ADR freezes
   seams and invariants, not unimplemented class names or a fictional final API.

## Compatibility rules

- Existing public imports and JSON envelopes remain compatible unless a later
  reviewed stage records an explicit additive change.
- Generic prepared-resource manifest version 1 stays the import contract.
  Export, when implemented by U1, must round-trip the same semantic graph; it
  must not create an incompatible second version-1 grammar.
- `Document.frontmatter` is the current parser-owned source metadata input.
  M1 must carry bounded normalized metadata through both `PreparedFile`
  construction paths into `ResourceRecord.metadata` without changing body,
  chunk, or embedding text.
- Exact metadata filters compile to existing facets and are applied before
  candidate limits. Runtime JSON scanning and range-query claims are excluded.
- Timed and frame data use the existing `mdrack-media` records, builders, and
  locators. Similarity remains explicitly textual.

## Consequences

B0 can open M1 without declaring 1.0 complete. Work already present in the
core/media/SQLite packages is not replayed. Missing behavior is assigned to one
application owner per stage, and any discovered need to change a frozen package
contract or schema becomes a separate decision gate rather than an incidental
1.1 edit.

## Non-claims

This decision does not claim final 1.0 or 1.1 readiness, a branch or tag, a
schema migration, a live provider, a private corpus, Windows, Python 3.12,
publication, or destructive cleanup.
