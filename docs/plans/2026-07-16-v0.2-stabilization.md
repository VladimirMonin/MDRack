# MDRack v0.2 Stabilization Implementation Plan

> **For Hermes:** execute this plan through the `mdrack` Kanban board with one
> writer at a time, independent review gates, and main-session Git closure.

**Goal:** Release MDRack v0.2 with stable indexing, provenance, assets, an
embedded Python API, and one shared FTS + semantic -> RRF retrieval path while
keeping production reranking explicitly deferred.

**Architecture:** CLI and `MDRackEngine` must call the same application-level
retrieval service over storage/search ports. Structural parsing remains lossless;
retrieval chunks are deterministic derived records. SQLite remains the only
persistent store, and LM Studio remains the only live embedding runtime.

**Tech stack:** Python 3.11+, uv, Click, Pydantic, markdown-it-py, SQLite/FTS5,
LM Studio HTTP API, pytest, Ruff.

**Canonical decisions:**

- [ADR-0001](../decisions/0001-reranking-deferred.md)
- [v0.2 modernization roadmap](../mdrack-v0.2-retrieval-modernization-plan.md)

---

## Release boundaries

### In scope

1. Historical/current baseline comparison.
2. One application-level retrieval path for CLI and embedded API.
3. Production-quality structural chunking and golden fixtures.
4. Public stable logical IDs and complete source locators.
5. Honest embedding dimensions/MRL capabilities and profile identity.
6. Local verification, migration/package smoke, documentation, commits and push.

### Out of scope

- production reranker adapters;
- `--rerank` or reranker configuration;
- a second `llama-server` runtime;
- chat-completion reranking emulation;
- cloud providers;
- direct Python model loading;
- new persistent databases.

## Phase 0 — Reranking status gate

**Status:** completed by commit `0708530`.

- ADR records model discovery separately from transport support.
- README documents the RRF-only production path.

## Phase 3 — Structural chunking quality

### Task 3.1: Freeze golden parser/chunker fixtures

**Files:**

- Create: `tests/fixtures/structural_chunking/`
- Create: `tests/unit/test_structural_chunking_golden.py`

Cover Russian/English Markdown, long paragraphs, nested headings, valid and
invalid Python, tables, Mermaid, wikilinks/embeds, Unicode and CRLF.

Golden assertions must prove:

- no empty chunks;
- no source text loss or duplicate ownership;
- stable source offsets, heading paths, logical IDs and asset references;
- identical logical chunks after repeated indexing.

### Task 3.2: Apply `min_chunk_chars` and safe small-block merging

**Files:**

- Modify: `src/mdrack/application/chunking.py`
- Modify: `src/mdrack/config/models.py`
- Test: `tests/unit/test_structural_chunking.py`

Merge only adjacent compatible prose/list/blockquote blocks. Never merge across a
heading, code fence, table, Mermaid block, image/embed, or thematic break.

### Task 3.3: Improve long prose and code splitting

**Files:**

- Modify: `src/mdrack/application/chunking.py`
- Test: `tests/unit/test_structural_chunking_golden.py`

Order for prose: paragraphs -> sentences -> deterministic word/character
fallback. Evaluate `semantic-text-splitter` only as a secondary long-prose
splitter; do not add it unless tests show a concrete benefit.

For Python use `ast` boundaries: module preamble, class, function and async
function. On syntax failure and for other languages, use complete-line windows.
Tables and Mermaid retain their dedicated policies.

## Phase 6 — Final release gate

Run:

```bash
UV_OFFLINE=1 bash scripts/verify.sh
uv run python scripts/check_no_forbidden_deps.py
uv build
```

Additionally verify:

- CLI/embedded parity E2E;
- fixed-corpus baseline report;
- SQLite migration and installed-wheel smoke;
- privacy scan of reports and logs;
- no source Markdown mutation;
- Windows PowerShell/EXE smoke on Windows, or explicitly report it unverified;
- real-vault audit only when the vault is readable; never substitute synthetic
  success for an unavailable source.

Commit boundaries:

1. `docs(reranking): defer production integration` — already created.
2. `test(baseline): add reproducible v0.2 comparison`.
3. `refactor(retrieval): unify CLI and embedded search`.
4. `feat(chunking): complete structural policies`.
5. `feat(provenance): expose stable logical identities`.
6. `feat(embeddings): report runtime dimensions honestly`.
7. `docs(v0.2): publish stabilization evidence`.

Each implementation commit must include focused tests and pass `git diff
--cached --check`. Main-session closure reviews every staged diff, runs the final
gates, pushes without force, and proves `HEAD == origin/master`.
