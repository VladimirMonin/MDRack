**Статус документа:** предлагаемый нормативный план реализации 1.1  
**Проект:** `VladimirMonin/MDRack`  
**Дата подготовки:** 19 июля 2026 года  
**Текущая подтверждённая ревизия репозитория:** `f7c829d5a3b48d5f24f2ec4d5c45a550ad7b8273` (`v0.3.1` offline evidence)  
**Входная продуктовая база:** завершённая версия 1.0 из предыдущего плана развития  
**Цель документа:** описать проверяемый путь от MDRack 1.0 к MDRack 1.1 без привязки ядра к Obsidian, Whisper, FFmpeg, LM Studio, OpenRouter, SQLite, WVM или иной конкретной host-системе.

---

## 0. Как читать этот документ

Этот план описывает **дельту 1.0 → 1.1**, а не повторяет весь roadmap становления проекта с текущей `0.3.1`.

При этом репозиторий на момент подготовки документа ещё не имеет опубликованного релиза `1.0`. Поэтому перед началом работ 1.1 обязателен входной gate:

```text
предыдущий план 1.0 выполнен
        ↓
контракты, пакеты, storage adapter и базовый retrieval проверены
        ↓
только после этого начинается функциональная дельта 1.1
```

Если какая-либо предпосылка 1.0 не выполнена, её нельзя незаметно «доделать по дороге». Она переносится в этап `0 — Entry Gate`, получает отдельный issue, тесты и commit boundary.

Термины:

- **MUST / обязательно** — release blocker;
- **SHOULD / рекомендуется** — допускается отклонение только через ADR;
- **MAY / опционально** — не влияет на готовность 1.1;
- **core** — пакет `mdrack-core`, не standalone-приложение;
- **application** — самостоятельный `mdrack` CLI/Python API и его integrations;
- **source metadata** — данные, полученные из исходника: YAML frontmatter, внешняя запись, media manifest;
- **facet** — нормализованное индексируемое точное значение для фильтрации;
- **timed atom** — мелкий сегмент распознавания с таймингами;
- **timed passage** — крупная смысловая поисковая единица, собранная из atoms;
- **frame caption** — текстовое описание кадра; в 1.1 это текстовый сигнал, не image embedding.

# 1. Резюме релиза 1.1

## 1.1. Формулировка версии

> **MDRack 1.1 — Universal Metadata and Text-Media Retrieval.**
>
> Самостоятельное и встраиваемое retrieval-приложение, которое индексирует типизированные ресурсы, сохраняет произвольную JSON-совместимую метаинформацию, поддерживает точные metadata filters, ищет по Markdown, timed transcripts и текстовым описаниям кадров и возвращает проверяемые source locators.

## 1.2. Что пользователь получает

В 1.1 пользователь сможет:

- индексировать Markdown с произвольным YAML frontmatter;
- не терять собственные вложенные metadata разных типов;
- настраивать, какие metadata становятся facets, title, aliases или отдельным lexical representation;
- фильтровать результаты по тегам, булевым, числовым и строковым scalar-полям через точное равенство;
- импортировать Whisper JSON, VTT, SRT и generic timed JSON;
- объединять мелкие speech-to-text segments в смысловые timed passages;
- искать по транскрипциям audio/video в text, semantic и hybrid режимах;
- получать `start_ms` и `end_ms` для перехода к найденному месту;
- добавлять текстовые описания редких кадров и ранжировать их как отдельные ветки;
- отдельно задавать вес речи, описаний кадров и metadata lexical signal;
- находить похожие документы/audio/video **по текстовому содержанию**;
- использовать Python API, CLI или generic resource manifest;
- подключать LM Studio, OpenAI-compatible endpoint или OpenRouter только во внешнем integration-слое;
- полностью работать offline с готовыми текстами и vectors.

## 1.3. Главная архитектурная гарантия

```text
extractors / providers / source applications
        │
        │  готовые тексты, тайминги, captions, vectors
        ▼
source projection / mdrack-media
        │
        │  PreparedResourceBatch
        ▼
mdrack-core
        │
        │  CatalogPort / SearchPort
        ▼
mdrack-sqlite или другой adapter
```

`mdrack-core` не должен знать:

- где находится файл;
- каким Whisper получена транскрипция;
- кто извлёк кадр;
- какой LLM написал caption;
- какой provider посчитал embedding;
- используется SQLite или PostgreSQL;
- является ли host-приложение MDRack, WVM или чем-либо ещё.

## 1.4. Прагматический принцип релиза

```text
сохранить данные без потери
→ индексировать только полезные проекции
→ использовать простые детерминированные алгоритмы
→ измерить качество
→ усложнять только после доказанной необходимости
```

В 1.1 не строится универсальный мультимодальный космический корабль. Все media-возможности остаются **text-first**.

# 2. Предпосылки версии 1.0

Перед реализацией 1.1 считаются доступными и стабильными следующие возможности.

## 2.1. Пакеты и зависимости

- `mdrack-core` устанавливается независимо;
- core не имеет provider-, filesystem-, HTTP-, Markdown- и persistence-зависимостей;
- `mdrack-sqlite` является отдельным adapter-пакетом либо эквивалентно отделённым distribution boundary;
- standalone `mdrack` зависит от core и adapters, но не наоборот;
- установленный wheel проверяется вне source tree;
- public exports имеют versioned contract.

## 2.2. Core domain

Доступны и считаются принятыми:

```text
ResourceRecord
RepresentationRecord
SearchUnitRecord
EmbeddingSpaceRecord
VectorRecord
ResourceFacet / Facet
Locator
PreparedResourceBatch
SearchRequest / SearchScope
LexicalBranch / VectorBranch
SearchResult / SimilarityResult
```

## 2.3. Retrieval

- lexical branch;
- vector branch;
- arbitrary multi-branch weighted RRF;
- scope до candidate limit;
- unit target;
- resource target;
- branch-local resource grouping;
- dense resource ranks после grouping;
- deterministic adapter-rank validation;
- safe degradation;
- whole-resource similarity по сохранённому vector;
- zero-vector policy;
- embedding-space fingerprint compatibility.

## 2.4. Persistence

- атомарный `replace_resource()`;
- атомарный и идемпотентный `delete_resource()`;
- metadata JSON round-trip;
- facets;
- lexical FTS;
- vector search;
- schema readiness;
- backup/rebuild/recovery procedure;
- adapter conformance suite.

## 2.5. Наблюдаемость и приватность

- raw query, content, metadata values, paths, provider bodies и vectors не логируются;
- request correlation ID безопасен и полезен;
- events содержат durations, counts, sizes, dimensions, branch categories и fingerprints;
- privacy sentinel suite является release gate.

## 2.6. Действия при невыполненной предпосылке

Если одна из перечисленных возможностей отсутствует:

1. создать issue категории `v1.0-entry-gap`;
2. добавить regression test, показывающий отсутствие возможности;
3. реализовать исправление отдельно от 1.1 feature work;
4. получить PASS полного v1.0 gate;
5. только затем разблокировать зависимые этапы 1.1.

## 4.3. Не плодить packages без необходимости

Для 1.1 достаточно:

```text

mdrack-media

```

Metadata projection не выделяется в отдельный PyPI-пакет. Она живёт в чистом importable модуле, например:

```text
mdrack.projections.metadata
```

Ограничения для этого модуля:

- без Click;
- без HTTP;
- без SQLite;
- без filesystem I/O;
- deterministic input → output;
- отдельные unit-тесты;
- может быть позднее выделен только после появления второго consumer-а.

## 4.4. Source of truth

- source files и host databases остаются источником истины;
- retrieval catalog является производным индексом;
- expensive extraction artifacts могут кэшироваться отдельно;
- core database не обязана содержать исходный media binary;
- удаление индекса не должно удалять источник;
- полная перестройка должна быть возможна из source + artifacts.

# 6. Универсальная модель metadata

## 6.1. Почему metadata нужна сейчас

Metadata — не Obsidian-специфичная функция. Она необходима для всех типов ресурсов:

```text
Markdown: tags, aliases, project, status, custom fields
Audio: language, duration, transcript origin
Video: channel, playlist, duration, capture source
Image: dimensions, source application, custom labels
External record: owner, tenant, workflow state, business keys
```

Фиксированные поля на каждый возможный source type разрушили бы универсальность. Правильная модель:

```text
канонические поля ресурса
+
непрозрачная JSON-compatible source metadata
+
явно проецируемые facets и lexical representations
```

## 6.2. Где хранить metadata

Полная source metadata хранится на уровне `ResourceRecord.metadata`, а не копируется в каждый chunk/unit.

Рекомендуемая envelope-структура:

```json
{
  "source": {
    "tags": ["rag", "python"],
    "draft": false,
    "priority": 3,
    "project": {
      "id": 42,
      "name": "MDRack"
    }
  },
  "ingestion": {
    "adapter": "markdown",
    "adapter_version": "1.1",
    "normalizer_version": "metadata-json-v1",
    "projection_policy_fingerprint": "sha256:..."
  },
  "derived": {
    "metadata_key_count": 5
  }
}
```

Правила:

- пользовательские значения помещаются только под `source`;
- internal fields не смешиваются с пользовательскими keys;
- `ingestion` содержит только безопасную техническую идентичность;
- `derived` не содержит сырые значения;
- `ResourceRecord.title` остаётся отдельным каноническим полем;
- `source_namespace`, `locator`, `content_hash`, `resource_kind`, `media_type` не дублируются в source metadata.

## 6.3. Поддерживаемые типы

Нормализатор 1.1 обязан сохранять различия между:

```text
string
integer
finite float
boolean
null
array
object
```

Пример:

```yaml
string_value: "3"
integer_value: 3
float_value: 3.0
flag: true
empty: null
list_value: [one, 2, false]
nested:
  key: value
```

После нормализации:

```json
{
  "string_value": "3",
  "integer_value": 3,
  "float_value": 3.0,
  "flag": true,
  "empty": null,
  "list_value": ["one", 2, false],
  "nested": {"key": "value"}
}
```

## 6.4. Нормализация YAML-specific типов

| YAML/Python значение | JSON projection |
|---|---|
| `date` | ISO `YYYY-MM-DD` string |
| `datetime` | ISO 8601 string с timezone, если он есть |
| tuple/set | ordered JSON array по deterministic policy |
| mapping с нестроковыми keys | keys преобразуются в strings либо field отклоняется |
| bytes/binary | отклоняется с diagnostic |
| NaN/Infinity | отклоняется с diagnostic |
| custom Python object/tag | отклоняется с diagnostic |
| duplicate YAML key | parser diagnostic; default fail metadata only, not document body |

Предпочтительная политика для set:

```text
нормализовать элементы
→ canonical JSON sort
→ сохранить как array
```

Это гарантирует детерминированность.

## 6.5. Limits

Default limits:

```toml
[metadata]
max_serialized_bytes = 65536
max_depth = 8
max_object_keys = 1000
max_array_items = 1000
max_string_bytes = 16384
invalid_policy = "warn_and_continue"
```

Поведение при превышении:

- основной документ продолжает индексироваться;
- metadata целиком или конкретное поле пропускается согласно diagnostic policy;
- raw values не попадают в лог;
- diagnostic содержит только category и counts;
- result/status сообщает, что metadata была частично отброшена.

## 6.6. Canonical serialization

- UTF-8;
- deterministic key ordering;
- finite numbers only;
- стабильное представление `-0.0` согласно принятому core contract;
- no implicit stringification неизвестных объектов;
- fingerprint вычисляется по canonical JSON bytes.

## 6.7. Metadata projection modes

Для каждого configured path поддерживаются режимы:

```text
store_only
canonical_title
facet
facet_many
lexical_text
ignore
```

Значение по умолчанию для неизвестного поля:

```text
store_only
```

Это означает: произвольные поля не теряются, но не влияют на поиск без явной настройки.

## 6.8. Path syntax

Использовать JSON Pointer-подобный синтаксис:

```text
/tags
/project/name
/custom/nested/value
```

Не использовать dotted path как канонический контракт, потому что пользовательский key может содержать точку.

Escape rules должны соответствовать JSON Pointer:

```text
~0 → ~
~1 → /
```

## 6.9. Пример projection config

```toml
[metadata]
max_serialized_bytes = 65536
invalid_policy = "warn_and_continue"

[[metadata.projections]]
path = "/title"
mode = "canonical_title"

[[metadata.projections]]
path = "/tags"
mode = "facet_many"
namespace = "tag"

[[metadata.projections]]
path = "/project/name"
mode = "facet"
namespace = "project"

[[metadata.projections]]
path = "/draft"
mode = "facet"
namespace = "draft"

[[metadata.projections]]
path = "/priority"
mode = "facet"
namespace = "priority"

[[metadata.projections]]
path = "/aliases"
mode = "lexical_text"

[[metadata.projections]]
path = "/private_plugin_state"
mode = "ignore"
```

## 6.10. Facet scalar codec

Текущий facet contract хранит string value. Чтобы различать типы без изменения core schema, projection layer использует canonical prefix codec:

```text
string  "3"   → s:3
integer 3     → i:3
float   3.0   → f:3.0
boolean true  → b:true
null          → z:null
```

Строки экранируются безопасным canonical codec; пользовательский CLI не обязан видеть prefix.

Преимущества:

- `"3"` не смешивается с `3`;
- schema core не меняется;
- SQLite/PostgreSQL adapters сравнивают обычные namespace/value;
- exact filter переносим;
- range semantics не притворяются поддержанными.

Ограничение:

- objects и arrays of objects не становятся facets;
- `facet_many` принимает только array scalars;
- числовые диапазоны не поддерживаются в 1.1.

## 6.11. Metadata lexical representation

Для aliases и выбранных human-readable fields создаётся отдельное representation:

```text
representation_kind = metadata_text
modality            = text
unit_kind           = whole_resource
```

Содержимое строится только из explicitly allowlisted fields:

```text
title
aliases
summary
selected custom labels
```

Оно:

- индексируется в FTS;
- по умолчанию не получает embedding;
- имеет небольшой отдельный branch weight;
- не смешивается с основным document body;
- не содержит status, timestamps, CSS classes и технические plugin values без явной настройки.

## 6.12. Что не реализуется

В 1.1 отсутствуют:

```text
priority >= 3
created between A and B
rating < 4.5
regex по metadata
arbitrary JSONPath
sorting по custom field
metadata aggregation/group-by
```

Для этого позднее может появиться `TypedAttribute`, но только после реального сценария и отдельного storage contract.

# 7. Markdown metadata pipeline

## 7.1. Data flow

```text
Markdown bytes
    ↓
YAML frontmatter parser
    ↓
JSON normalizer
    ↓
source metadata
    ├── ResourceRecord.metadata.source
    ├── canonical title
    ├── ResourceFacet[]
    └── optional metadata_text representation

Markdown body
    ↓
structural parser/chunker
    ↓
retrieval text units
```

## 7.2. Главный инвариант

Frontmatter не должен автоматически попадать:

- в body chunk text;
- в основной embedding text;
- в snippets;
- в каждый SearchUnit metadata;
- в logs.

## 7.3. Default Obsidian policy

Без пользовательской настройки:

```text
title   → ResourceRecord.title
tags    → facet_many namespace=tag
aliases → metadata_text lexical representation
остальное → store_only
```

Это минимально полезный default, но core не знает слова Obsidian.

## 7.4. Custom metadata

Пользователь может хранить:

```yaml
---
reviewed: false
priority: 4
project:
  code: MDR
  phase: implementation
people:
  - Alice
  - Bob
custom_plugin:
  state:
    cursor: 17
---
```

Вся структура сохраняется. Фильтруемыми становятся только поля из projection config.

## 7.5. Ошибочный frontmatter

Default:

```text
ошибка frontmatter
→ diagnostic METADATA_PARSE_FAILED
→ body индексируется
→ metadata отсутствует
→ команда завершается success-with-diagnostics
```

Strict mode:

```toml
[metadata]
invalid_policy = "fail_resource"
```

используется только явно.

## 7.6. Reindex behavior

Изменение frontmatter меняет source hash и инициирует обычный replace resource graph.

Изменение projection policy также должно инициировать rebuild даже при неизменном source file. Для этого readiness/index state хранит:

```text

metadata_projection_policy_fingerprint
```

## 8.1. TimedTextAtom

## 8.4. Timeline locators

Нормативные forms:

```python
Locator(

        "start_ms": 120_000,

    },

```

```python
Locator(
    "video_frame",
    {

```

Правила:

- integer milliseconds only;
- interval semi-open `[start_ms, end_ms)`;
- `start_ms >= 0`;
- `end_ms > start_ms`;
- frame timestamp non-negative;
- typed helper validates payload before conversion to generic core `Locator`.

## 8.7. Связь passages и кадров

Не создавать жёсткое поле:

```text
passage.frames[]
```

```text
passage: [120000, 180000)

```

Presentation/host layer может выбрать:

- все кадры внутри interval;
- ближайший к центру;
- ближайший перед start;
- top-N frame-caption evidence.

Это не responsibility core.

## 8.8. Whole-resource textual representation

Используется для поиска похожих ресурсов по содержанию текста.

Первый baseline:

```text
embed full normalized text directly
```

```text
similarity_basis = textual_content
aggregation = token_weighted_centroid_v1 | direct_text_v1
```

Нельзя называть это visual/audio similarity.

# 9. Timed passage chunking

## 9.1. Задача

Преобразовать мелкие и нестабильные speech-to-text atoms в passages, которые:

- содержат достаточно контекста для embeddings;
- не объединяют слишком много тем;
- имеют точные тайминги;
- детерминированы;
- не требуют LLM;
- не зависят от Whisper-specific классов.

## 9.2. Reference policy

```python
TimedChunkingPolicy(
    minimum_tokens=160,
    target_tokens=320,
    soft_max_tokens=480,
    hard_max_tokens=800,
    target_duration_ms=60_000,
    soft_max_duration_ms=90_000,
    hard_max_duration_ms=120_000,
    pause_boundary_ms=800,
    overlap_atoms=0,
)
```

Это baseline для evaluation, а не вечная истина.

## 9.3. Три профиля для эксперимента

| Профиль | Target tokens | Soft max | Hard max | Target duration | Hard max duration |
|---|---:|---:|---:|---:|---:|
| compact | 180 | 320 | 480 | 40 s | 75 s |
| balanced | 320 | 480 | 800 | 60 s | 120 s |
| large | 480 | 800 | 1200 | 90 s | 180 s |

Default 1.1: `balanced`.

## 9.4. Boundary signals

В порядке приоритета:

1. конец предложения;
2. смена speaker;
3. pause `>= pause_boundary_ms`;
4. chapter/source boundary;
5. soft max;
6. hard max.

## 9.5. Детерминированный алгоритм

```text
current = []

for atom in ordered_atoms:
    validate atom
    append atom

    if current below minimum:
        continue

    if good natural boundary and target reached:
        emit passage
        continue

    if soft max reached:
        emit at best available atom boundary
        continue

    if hard max reached:
        emit immediately

emit remainder if non-empty
```

Точный выбор boundary среди последних atoms должен быть deterministic: одинаковый input и policy создают одинаковые passages и IDs.

## 9.6. Работа без пунктуации

Если punctuation отсутствует:

```text
minimum reached
→ использовать pause/speaker boundary
→ после soft max закрыть на atom boundary
→ после hard max закрыть обязательно
```

Не использовать LLM для расстановки точек как часть core pipeline.

## 9.7. Oversized atom

Если один atom превышает hard max:

- при word timestamps: разбить на word boundary с точными subranges;
- при segment timestamps: предпочтительно оставить oversized atom единым passage;
- опциональный emergency text split допустим только с `timing_precision=approximate`;
- emergency split должен быть явно посчитан в diagnostics;
- silent proportional timing inference запрещён.

## 9.8. Overlap

Default:

```text
overlap_atoms = 0
```

Overlap может быть включён только после ablation test, если:

- Recall@5 статистически/практически улучшается;
- duplicate top-k rate остаётся в принятом бюджете;
- storage/vector growth приемлем;
- timestamp evidence остаётся понятным.

## 9.9. TokenCounter

`mdrack-media` зависит от protocol:

```python
class TokenCounter(Protocol):
    def count(self, text: str) -> int: ...
```

Reference implementations:

- exact provider tokenizer в application layer;
- deterministic estimated counter для offline tests;
- whitespace count не называется exact tokens.

Каждый representation/unit хранит:

```text
token_count
token_count_kind = exact | estimated
```

## 9.10. Chunker fingerprint

Fingerprint включает:

- algorithm version;
- policy values;
- token counter name/version;
- normalization version;
- punctuation rules version;
- language-independent/optional language profile.

Изменение fingerprint требует rebuild соответствующих resources.

# 10. Ranking и search orchestration

## 10.1. Ветки видео

```text
transcript lexical
transcript semantic
frame-caption lexical
frame-caption semantic
metadata lexical
```

## 10.2. Branch-local scope

Каждая branch должна уметь дополнительно ограничить representation/unit:

```python
LexicalBranch(
    branch_id="transcript_lexical",
    query=query,
    weight=0.4,
    candidate_limit=100,
    scope_override=SearchScope(
        representation_kinds=("transcript_text",),
        unit_kinds=("time_segment",),
    ),
)
```

Эффективный scope:

```text
request.scope ∩ branch.scope_override
```

Правила intersection должны быть документированы и протестированы для всех filter fields.

## 10.3. Default weights

```toml
[search.video]
transcript_weight = 1.0
frame_caption_weight = 0.5
metadata_weight = 0.2
lexical_fraction = 0.4
semantic_fraction = 0.6
```

Производные веса:

```text
transcript lexical       0.40
transcript semantic      0.60
frame-caption lexical    0.20
frame-caption semantic   0.30
metadata lexical         0.20
```

Metadata semantic branch по умолчанию отсутствует.

## 10.4. Presets

```text
speech_first
balanced
frames_first
```

Пример:

| Preset | Transcript | Frame captions | Metadata |
|---|---:|---:|---:|
| speech_first | 1.0 | 0.35 | 0.15 |
| balanced | 1.0 | 1.0 | 0.20 |
| frames_first | 0.6 | 1.0 | 0.15 |

Автоматический query classifier не входит в 1.1.

## 10.5. Candidate limits

Default:

```text
candidate_limit_per_branch = max(100, result_limit * 10)
```

Для frame-caption branch допускается больший limit из-за resource crowding, но он должен быть bounded config.

## 10.6. Resource grouping

В каждой branch:

1. adapter возвращает units;
2. core валидирует positional ranks;
3. units группируются по `resource_id`;
4. выбирается representative candidate;
5. resource ranks перенумеровываются `1..N`;
6. только затем выполняется RRF.

Один ресурс получает не более одного RRF contribution от одной branch независимо от числа passages/frames.

## 10.7. Resource crowding regression

Обязательный fixture:

```text
video A: 90 высоких frame-caption units
video B: 2 хороших units
video C: 1 хороший unit
```

Проверяется:

- A не получает 90 голосов;
- B/C не исчезают из resource results только из-за unit crowding в разумном candidate budget;
- evidence limit соблюдается;
- dense ranks корректны.

Если обычного overfetch недостаточно, adapter-level diversification оформляется отдельным ADR. Нельзя незаметно менять port contract.

## 10.8. Добавление новых кадров

В 1.1 используется full resource replacement:

```text
load cached transcript artifacts
load existing frame captions
add new frame captions
rebuild complete PreparedResourceBatch
replace_resource atomically
```

Плюсы:

- нет partial graph;
- простая recovery;
- один production writer algorithm;
- изменившийся frame policy полностью воспроизводим.

Incremental unit patch API откладывается до измеренной необходимости.

## 10.9. Metadata filters

CLI/API exact filters преобразуются в facets до вызова core:

```text
--tag rag
--meta draft=false
--meta priority=3
--meta project=MDRack
```

Filters применяются adapter-ом до candidate limit.

## 10.10. Result evidence

Timed result должен содержать публично:

```text
resource_id
unit_id
score
rank
representation_kind
start_ms
end_ms
frame timestamp — если frame unit
source locator
branch evidence
text preview — только намеренный result payload
```

Logs не содержат preview.

OpenRouter не является частью core и не обязателен для релиза.

- API key только через environment/secret store;
- zero automatic network calls;
- no provider retry without configured policy;
- request/response body не логируются;
- paid E2E запускается только вручную;

Кэш расположен вне core catalog.

Key:

```text
source content fingerprint
+
producer/model fingerprint
+
prompt/config fingerprint
+
preprocessing fingerprint
```

Кэшируемые artifacts:

- normalized transcript atoms;
- timed passages;
- frame manifest;
- frame captions;
- text vectors;
- whole-resource vectors;
- metadata normalization/projection result — optional.

Cache rules:

- immutable entries;
- content-addressed directories или key-value store;
- safe metadata only in logs;
- explicit purge command;
- corrupted entry discarded and rebuilt;
- cache miss не изменяет core semantics.

```text
provider_auth_failed
invalid_provider_response

caption_count_mismatch
artifact_cache_corrupt
```

Raw exception message не является public/log contract.

# 12. Persistence и migration strategy

## 12.1. Требования к storage adapter

`mdrack-sqlite` должен сохранять:

- resource metadata canonical JSON;
- representations;
- units и locators;
- embedding spaces/fingerprints;
- vectors;
- facets;
- FTS projection;
- readiness/version information.

## 12.2. Новые schema needs 1.1

Если v1.0 schema уже содержит `metadata_json` и facets, новый core schema не требуется.

Возможные additions:

- index на facet `(namespace, value, resource_id)`;
- representation/unit indexes для branch scope;
- optional `metadata_fingerprint` в resource row или readiness record;
- optional projection policy fingerprint в resource metadata, не отдельной колонкой.

Нельзя добавлять таблицу произвольных JSON attributes только ради будущих range queries.

## 12.3. Metadata exact filter

Filters используют facets, не runtime JSON scan.

Плохой путь:

```text
load every metadata_json
→ parse in Python
→ filter after top-k
```

Правильный путь:

```text
projection creates facets
→ indexed SQL joins
→ scope before FTS/vector limit
```

## 12.4. Migration compatibility

1. v1.0 database открывается read-only validation;
2. 1.1 migration создаёт только необходимые indexes/schema additions;
3. existing resources получают `metadata={}` semantics;
4. source resources должны быть re-ingested для появления metadata/facets;
5. migration не выдумывает frontmatter из старых chunks;
6. backup/rollback документируются;
7. rebuild mode предпочтительнее risky in-place backfill.

## 12.5. Rebuild states

```text
ready_v1_0
→ rebuild_required_v1_1
→ building_v1_1
→ ready_v1_1
             ↘ failed_v1_1
```

Причины rebuild:

- metadata normalizer version;
- metadata projection policy fingerprint;
- timed chunker fingerprint;
- embedding profile fingerprint;
- FTS projection version.

## 12.6. PostgreSQL/pgvector

Не реализуется в 1.1.

Обязательная подготовка:

- conformance tests не используют SQLite-specific API;
- core ports остаются стабильными;
- manifest/DTO не содержат SQLite row IDs;
- filters выражаются через `SearchScope`/facets;
- benchmark собирает operating envelope.

Решение о pgvector принимается после 1.1 production-like measurements.

# 27. Migration, recovery и rollback

## 27.1. Upgrade strategy

Предпочтительно:

```text
v1.0 active catalog
→ create v1.1 candidate catalog
→ re-ingest/rebuild resources
→ verify
→ atomic active pointer switch
→ retain v1.0 catalog for rollback window
```

## 27.2. Почему rebuild предпочтительнее backfill

Metadata и timed passages зависят от source artifacts и policies, которых нет в старом index. Backfill из chunks не может восстановить:

- raw frontmatter;
- custom types;
- source atoms;
- exact timings;
- frame captions;
- new fingerprints.

## 27.3. Verification candidate

- schema integrity;
- foreign keys;
- canonical JSON;
- facet count/codec;
- FTS row count;
- vector dimensions/fingerprint;
- resource/representation/unit relationships;
- locator validation;
- readiness fingerprints;
- sample searches;
- no privacy data in diagnostics.

## 27.4. Rollback

Rollback:

- переключает active pointer на v1.0;
- не удаляет v1.1 candidate;
- не пытается открыть future schema старым binary;
- не удаляет source/cache;
- документируется отдельной command/procedure.

## 27.5. Cleanup

Destructive cleanup отдельна:

```bash
mdrack cache purge --confirm
mdrack resource cleanup-generations --older-than ... --confirm
```

Никакой automatic cleanup в release cutover.

Риск: Whisper atoms дают плохой context и много vectors.

Защита:

- timed grouper;
- balanced default;
- chunk ablation;
- passages/hour metrics.

## 28.5. Слишком крупные passages

Риск: embedding усредняет много тем.

Защита:

- token + time limits;
- natural boundaries;
- quality/timestamp metrics.

## 28.6. Плохие тайминги

Риск: split внутри segment создаёт ложную точность.

Защита:

- atom-boundary split;
- word timings when available;
- approximate marker;
- no silent interpolation.

## 28.7. Длинные видео доминируют

Риск: много passages/frames вытесняют другие resources.

Защита:

- branch-local grouping;
- dense resource ranks;
- one contribution/resource/branch;
- crowding fixture;
- bounded overfetch.

## 28.8. Frame captions ухудшают speech search

Защита:

- separate branches;
- `speech_first` default;
- ablation;
- configurable weights;
- branch disable.

## 28.9. Provider lock-in

Защита:

- protocols;
- ready vector import;
- fingerprints;
- generic manifest;
- cache outside core;
- optional adapters.

## 28.10. SQLite становится медленным

Защита:

- measured envelope;
- frame limits;
- passages not atoms;
- adapter boundary;
- post-1.1 backend ADR.

## 28.11. Privacy leak через metadata

Защита:

- metadata values never logged;
- safe diagnostics;
- separate public `metadata show`;
- sentinel assets;
- no raw metadata in evidence.

## 28.12. WVM диктует core schema

Защита:

- WVM-like compatibility proof only after 1.1;
- source locator generic;
- host adapter maps own DB;
- no WVM table/ID names in core.

# 33. Финальная архитектурная формула

```text
Markdown/YAML, Whisper JSON, VTT/SRT, frame manifests
        │
        ▼
source readers and projection policies
        │
        ├── raw JSON metadata
        ├── facets
        ├── timed atoms/passages
        ├── frame captions
        └── ready text vectors
        ▼
PreparedResourceBatch
        ▼
mdrack-core
        ├── validation
        ├── filtered branches
        ├── resource grouping
        ├── weighted RRF
        └── textual similarity
        ▼
mdrack-sqlite
        ▼
standalone CLI/Python API или будущий host adapter
```

Ключевые решения 1.1:

```text
metadata сохраняется полностью, но индексируется только явно
custom types не сводятся к одной строке
Whisper atom не является retrieval passage
passage имеет точный временной диапазон
кадр и passage связаны timeline, а не жёсткой вложенностью
frame caption — отдельная текстовая branch
речь по умолчанию весит больше кадров
новые кадры заменяют полный resource graph атомарно
similarity audio/video в 1.1 означает textual similarity
providers остаются внешними
core остаётся переносимым
```

Этот план доводит MDRack от стабильного универсального 1.0 к практически полезному 1.1, который сохраняет пользовательские metadata, работает с текстовыми представлениями аудио и видео, остаётся самостоятельным приложением и при этом не приобретает зависимость от будущей схемы Whisper Voice Machine или любого другого конкретного проекта.
