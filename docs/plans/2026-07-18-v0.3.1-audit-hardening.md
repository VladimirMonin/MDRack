# MDRack v0.3.1 — audit hardening plan

Статус: proposed, implementation-ready
Основание: внешний аудит `master` на `e4cdc7aab7baa42fd116adf53f40a6b0dd7a112a` и локальная проверка исходников 2026-07-18.

## 1. Вывод

Архитектура v0.3 состоялась, но перед расширением на большие мультимодальные ресурсы нужно закрыть несколько контрактных дыр. Самый высокий риск — ранжирование ресурсов после группировки и отказ всей cosine-ветки из-за одного нулевого candidate vector.

Подтверждённые дефекты нельзя смешивать с осознанными ограничениями. Отдельный `mdrack-core` distribution, PostgreSQL/ANN, audio/video ingestion и production reranking не входят в v0.3.1.

## 2. Проверенные замечания

| Замечание | Статус | Основание |
|---|---|---|
| Resource rank не уплотняется после grouping | **подтверждено, P0** | `_group_resources()` сохраняет `representative.rank`; `weighted_rrf()` использует этот старый rank |
| Один нулевой cosine candidate ломает ветку | **подтверждено, P0** | SQLite `_score()` выбрасывает `INCOMPATIBLE_VECTOR_SPACE`; поведение закреплено интеграционным тестом |
| Adapter rank contract доверяет rank | **подтверждено, P0** | `_validate_candidates()` проверяет только list/type/branch_id |
| Query branch не несёт fingerprint пространства | **подтверждено, P1** | `VectorBranch` содержит `space_id`, vector, weight и limit, но не fingerprint |
| Настройки `text_weight`/`semantic_weight` не применяются | **подтверждено, P1** | стандартный `search_hybrid()` создаёт обе ветки с default `weight=1.0` |
| Image semantic `score` — RRF, а не raw cosine | **подтверждено, P1** | `_map_search()` публикует `SearchResultItem.score`; raw score остаётся в evidence |
| Нет высокоуровневого visual query path | **подтверждено, feature gap** | visual vectors индексируются, но query branch всегда использует `text_space` |
| `max_text_tokens` считается через `split()` | **подтверждено, P2** | `_estimated_tokens()` используется как жёсткий предел whole-resource text |
| Markdown scan не создаёт whole-resource vector | **подтверждено, feature gap** | стандартная проекция создаёт chunk units; whole-resource пример есть только в generic paths/tests |
| Качество semantic retrieval на real embeddings не доказано | **подтверждено, evidence gap** | release gate использует fakes/fixtures и честно исключает live LM Studio/vault |
| SQLite vector search — O(n) Python scan | **подтверждено, documented limit** | все подходящие JSON vectors загружаются и сортируются в Python |

## 3. Stage A — корректное resource-level ранжирование (P0)

### Изменение

После branch-local grouping назначать ресурсам новые плотные ранги `1..N` в порядке первого лучшего представителя. Исходный unit rank сохранять только в evidence. Weighted RRF должен использовать плотный resource rank.

### Контракт adapter-а

До grouping валидировать результат search port:

- список уже упорядочен по rank;
- ranks — точная последовательность `1..len(result)`;
- позиция элемента соответствует rank;
- `branch_id` совпадает;
- `unit_id` не повторяется;
- `candidate_limit` соблюдён adapter-ом, а не только core slicing.

Некорректный backend-result преобразуется в безопасный `ADAPTER_ERROR`; приватные значения не логируются.

### Тесты

- `A,A,A,B` с unit ranks `1,2,3,4` превращается в resource ranks `A=1,B=2`;
- длинный video-like resource с 500 units не сдвигает следующий ресурс до rank 501;
- одинаковый результат Memory/SQLite/fake malformed adapter;
- shuffled, duplicate, missing и non-contiguous ranks fail closed;
- unit-level target сохраняет прежнее поведение.

### Done

RRF использует resource rank после grouping; unit rank виден только как evidence. Полный suite и installed-package gate зелёные.

## 4. Stage B — устойчивость vectors и identity пространства (P0/P1)

### Нулевые vectors

Политика:

1. Нулевой cosine **query vector** остаётся `INCOMPATIBLE_VECTOR_SPACE`.
2. Новые нулевые cosine vectors отклоняются в `CoreIndexingService` до записи.
3. Старый/повреждённый нулевой candidate не должен валить всю ветку: adapter пропускает candidate, считает безопасный `degenerate_candidate_count` и продолжает.
4. Если допустимых candidates не осталось, ветка возвращает пустой результат с безопасной degradation-категорией, не raw exception.
5. Dot/L2 zero vector остаётся допустимым, если контракт метрики это разрешает.

### Fingerprint

Расширить `VectorBranch` полем `expected_space_fingerprint`:

- application paths всегда передают fingerprint;
- adapter сравнивает его с `EmbeddingSpaceRecord.fingerprint`;
- mismatch даёт `INCOMPATIBLE_VECTOR_SPACE` до scan;
- временно `None` допустим только для совместимости direct core consumers и помечается в документации как deprecated loose mode.

### Тесты

- 10 000 нормальных + один zero candidate не выключают search;
- zero query fail closed;
- cosine/dot/l2 policy matrix;
- одинаковая размерность, но другой fingerprint отклоняется;
- Memory/SQLite parity и corruption fixtures.

## 5. Stage C — публичная hybrid/score семантика (P1)

### Branch weights

Подключить `SearchConfig.text_weight` и `semantic_weight` к `LexicalBranch.weight` и `VectorBranch.weight` во всех стандартных CLI/Engine paths. Валидировать, что веса конечны, неотрицательны и хотя бы один положителен. Удалить документацию об их неиспользовании.

### Scores

Не перегружать одно поле разными смыслами:

- semantic result `score` = raw metric score;
- hybrid result `score` = RRF/fusion score;
- дополнительно сохранять явные `semantic_score`, `text_score`, `fusion_score` там, где DTO уже допускает расширение;
- image semantic mapping следует той же политике, что Markdown compatibility mapper;
- evidence остаётся диагностикой, а не единственным местом raw semantic score.

### Тесты

- веса `0.9/0.1` и `0.1/0.9` меняют предсказуемый fixture-order;
- CLI и MDRackEngine byte-parity;
- Markdown/image semantic score parity;
- schema/compatibility и installed-wheel tests.

## 6. Stage D — product gaps без смешивания с core fixes (P2)

### D1. Visual similarity API

Добавить явные операции:

- query image → visual vector → image similarity;
- existing image resource → stored visual whole-resource vector → similarity;
- text → visual space только при явно заявленном shared multimodal fingerprint.

Не подменять visual search OCR/caption text search. В DTO возвращать используемые modality и space identity.

### D2. Text budget

Заменить вводящее в заблуждение `max_text_tokens`:

- ввести injectable `TextBudgetEstimator`/token counter;
- LM Studio profile может предоставлять tokenizer-aware estimator;
- fallback явно называется `estimated_units`, не tokens;
- всегда иметь независимые hard byte/char limits;
- покрыть русский, CJK, JSON, code, punctuation и длинные identifiers.

### D3. Markdown whole-resource similarity

Сначала принять продуктовую политику, потому что она увеличивает embedding cost:

- configurable document-level representation/unit;
- один `UNIT_WHOLE_RESOURCE` vector на Markdown resource;
- стабильный ID и fingerprint;
- replace/delete/rebuild lifecycle;
- similarity API для документов и смешанного resource scope.

По умолчанию feature можно оставить выключенной до замера стоимости и качества.

## 7. Stage E — доказательство качества semantic retrieval

Механику нельзя выдавать за relevance quality. Нужен отдельный повторяемый benchmark:

- фиксированный публичный или синтетический corpus без личных данных;
- минимум русский + английский + code/Markdown;
- зафиксированные queries и relevance judgments;
- реальные LM Studio embeddings для явно названных моделей/profile/fingerprint;
- baseline: lexical, semantic, hybrid unweighted, hybrid weighted;
- метрики: Recall@5/10, MRR@10, nDCG@10, latency p50/p95, index time, vector count;
- raw private query/content не попадают в release logs;
- результат `tested/not installed/unsupported`, без предполагаемых галочек.

Live benchmark не блокирует unit correctness release, но блокирует утверждения о качестве semantic search.

## 8. Stage F — performance envelope

SQLite O(n) остаётся допустимым backend-ом v0.3.1, но нужно измерить предел:

- fixtures 10k/50k/100k vectors для 384/768/1024 dimensions;
- latency и peak RSS;
- documented support envelope;
- warning/diagnostic при превышении проверенного диапазона;
- отдельное решение об ANN/PostgreSQL только после измерений и появления второго backend consumer.

## 9. Дополнительные hardening-задачи

- Ввести schema validators для известных locator kinds (`time_segment`, `frame`, `region`) на ingestion boundary; generic core JSON locator оставить расширяемым.
- Добавить regression на `start_ms < end_ms`, frame index/range и region bounds.
- Отдельный `mdrack-core` distribution оставить deferred до второго реального consumer-а.
- Audio/video demux, STT, frame sampling и captioning оставить вне v0.3.1; использовать их только как stress fixtures для resource ranking.
- Production reranking остаётся deferred: никакого chat/completions или несовместимого llama-server workaround.

## 10. Порядок commits

1. `fix(core): compact resource ranks before weighted fusion`
2. `fix(core): validate adapter rank contracts`
3. `fix(search): isolate degenerate cosine candidates`
4. `feat(core): bind vector queries to space fingerprints`
5. `fix(app): apply configured hybrid branch weights`
6. `fix(images): expose consistent semantic score contracts`
7. `feat(images): add visual similarity query paths`
8. `fix(images): replace word-count token limits with explicit budgets`
9. `feat(indexing): add optional document whole-resource vectors`
10. `test(retrieval): add live semantic quality benchmark harness`
11. `docs(v0.3.1): publish measured retrieval and performance envelope`

Каждый commit проходит focused tests, full suite, Ruff, dependency/core guards и privacy checks. Изменения core contracts требуют Memory/SQLite parity и installed-package проверки.

## 11. Release acceptance

v0.3.1 принимается, когда:

- resource grouping не искажает branch ranks;
- один degenerate candidate не выключает semantic search;
- backend rank contract проверяется fail closed;
- app queries связаны с fingerprint пространства;
- configured hybrid weights реально действуют;
- Markdown и image score semantics согласованы;
- visual/document similarity честно помечены как implemented либо deferred;
- release evidence различает correctness, offline E2E, live component checks и relevance benchmark;
- suite, package builds, isolated installs, privacy и zero-unexpected-network gates проходят.
