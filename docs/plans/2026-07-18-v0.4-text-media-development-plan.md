# MDRack: подробный план развития самостоятельного retrieval-ядра и текстового media-поиска

**Статус документа:** предлагаемый рабочий план  
**Проект:** `VladimirMonin/MDRack`  
**Проверенная базовая ревизия:** `e4cdc7aab7baa42fd116adf53f40a6b0dd7a112a`  
**Дата подготовки:** 18 июля 2026 года  
**Назначение:** развитие MDRack как самостоятельной библиотеки и приложения, пригодных для повторного использования в других проектах; будущая интеграция с Whisper Voice Machine рассматривается как проверка универсальности, но не диктует архитектуру.

---

## 0. Краткое резюме решения

MDRack следует развивать как четыре изолированных слоя:

```text
готовые тексты, тайминги и текстовые векторы
                    │
                    ▼
          mdrack-media (pure Python)
  временные атомы → смысловые timed passages
  кадры/описания кадров → типизированные units
  подготовка PreparedResourceBatch
                    │
                    ▼
           mdrack-core (pure kernel)
  Resource / Representation / SearchUnit / Vector
  validation / filtered retrieval / grouping / RRF
  similarity / safe observability / ports
                    │
                    ▼
            storage adapter
  mdrack-sqlite сейчас
  mdrack-pgvector только после измерений
                    │
                    ▼
              host application
  standalone MDRack / будущий WVM / другой проект
```

В ближайшем roadmap используются **только текстовые представления**:

- Markdown-текст;
- OCR или текстовое описание изображения;
- транскрипция аудио;
- транскрипция аудиодорожки видео;
- текстовые описания отдельных кадров;
- обычные text embeddings этих текстов.

Нативные audio embeddings, image embeddings, video embeddings, сравнение по голосу, музыке, пикселям или визуальной композиции **не входят в текущий план**. Существующие абстракции embedding spaces сохраняют техническую возможность добавить их позже, но ни один текущий этап, тест или критерий готовности от них не зависит.

Основной retrieval unit для речи — не мелкий Whisper segment, а **детерминированный смысловой временной фрагмент** с `start_ms`, `end_ms`, текстом и стабильным ID. Сырые Whisper/VTT/SRT сегменты остаются входными атомами внешнего слоя и напрямую не индексируются.

Кадры и транскрипционные фрагменты не вкладываются друг в друга. Они принадлежат одной timeline и связываются по времени:

```text
timed passage: [120000, 180000)
frame:                    153000
```

Такой подход остаётся универсальным и не привязывает ядро к текущей или будущей схеме базы WVM.

## 1.2. Что пока остаётся переходным

На базовой ревизии:

- `mdrack_core` физически отделён по imports, но поставляется внутри общего wheel `mdrack`;
- общий wheel всё ещё устанавливает Click, HTTPX, Pydantic, Markdown parser и app-зависимости;
- generic SQLite schema существует как migration `0007` после legacy Markdown migrations;
- standalone reusable SQLite package отсутствует;
- текущий semantic search SQLite выполняет линейный Python scan JSON-векторов;
- current app одновременно поддерживает legacy и core-backed compatibility surfaces;
- media projection toolkit отсутствует;
- whole-resource vectors для обычных Markdown-документов стандартным pipeline не создаются;
- качество retrieval проверено в основном deterministic fake vectors, а не размеченным реальным корпусом.

## 1.3. Обязательные исправления до нового roadmap

Замечания предыдущего аудита считаются входным долгом:

1. пересчитать branch-local resource ranks после группировки units;
2. определить корректную zero-vector policy;
3. защититься от передачи query vector из другого embedding fingerprint;
4. нормализовать и валидировать ranks, возвращённые adapter-ом;
5. унифицировать публичную семантику `score`;
6. добавить стандартный whole-resource vector path для документов;
7. подключить реальные hybrid branch weights в app surface;
8. сделать безопасный корреляционный `request_id`;
9. включить mypy в release gate либо честно убрать ложное обещание strict typing;
10. проверить high-level текстовый поиск и similarity изображений без переоценки visual-возможностей.

Новые media-контракты не должны строиться до закрытия этих пунктов и фиксации стабильного core contract.

# 2. Цели проекта

## 2.1. Главная цель

Создать самостоятельный retrieval stack, который можно:

- установить как библиотеку;
- использовать из Python;
- запускать как standalone CLI-приложение;
- подключить к любому host-приложению через adapters;
- перенести с SQLite на другой backend без изменения retrieval-логики;
- кормить любыми заранее подготовленными текстами, таймингами и векторами;
- использовать для Markdown, изображений, аудио и видео без знания конкретных extractor-ов.

## 2.2. Продуктовые цели ближайших версий

1. Надёжный text, semantic и hybrid search по Markdown.
2. Прямое индексирование изображений через OCR/caption text.
3. Generic prepared-resource import.
4. Поиск по транскрипциям аудио.
5. Поиск по транскрипциям видео.
6. Поиск по текстовым описаниям кадров.
7. Возврат точных временных диапазонов и frame timestamps.
8. Поиск похожих ресурсов **по их текстовому содержанию**.
9. Полностью автономный SQLite deployment.
10. Чёткий путь к будущему PostgreSQL/pgvector adapter-у без его преждевременной реализации.

## 2.3. Архитектурные цели

- Core не вызывает LLM, VLM, OCR, Whisper или embedding provider.
- Core не открывает файлы и сеть.
- Core не импортирует SQLite/PostgreSQL.
- Core не знает Markdown, аудио, видео или WVM workflow.
- Core знает типизированные ресурсы, representations, units, locators, vector spaces, filters и ranking.
- Media projection остаётся provider-neutral и storage-neutral.
- Host владеет source lifecycle и дорогими extraction artifacts.
- Storage index считается производным и перестраиваемым.

# 3. Явные нецели текущего roadmap

Следующее не реализуется в рамках данного плана:

- audio embeddings;
- image embeddings;
- visual similarity по пикселям;
- acoustic similarity по голосу, шуму или музыке;
- единый multimodal vector;
- video embeddings;
- object detection;
- image regions и bounding boxes;
- покадровое индексирование всего видео;
- diarization как обязательная функция;
- speaker recognition;
- daemon или распределённый worker;
- cloud-first deployment;
- PostgreSQL adapter до benchmark gate;
- ANN-индекс до benchmark gate;
- автоматическая интеграция с WVM;
- знание схемы базы WVM внутри MDRack;
- автоматическое сканирование картинок, упомянутых в Markdown;
- генерация конечного RAG-ответа;
- хранение бинарных аудио, видео и изображений в retrieval DB.

Существующий experimental visual-vector path можно сохранить ради совместимости, но он:

- не расширяется;
- не входит в public acceptance;
- не влияет на приоритеты;
- не используется как доказательство visual search.

# 5. Универсальная модель сущностей

## 5.1. Resource

Один пользовательский исходный объект:

```text
document
image
audio
video
```

Примеры:

```text
один Markdown-файл
одна PNG-картинка
один WAV/MP3-файл
один MP4-файл
```

Audio track, извлечённый из видео, не становится отдельным Resource, пока пользователь не воспринимает его как отдельный объект.

Кадр, извлечённый из видео, также остаётся unit внутри Video Resource, а не отдельным Image Resource.

## 5.2. Representation

Способ представить содержание ресурса:

```text
retrieval_text
ocr_text
caption_text
transcript_text
audio_transcript
frame_caption
```

Один Video Resource может иметь:

```text
transcript representation
frame-caption representation
whole-transcript representation
```

## 5.3. SearchUnit

Минимальная ранжируемая единица:

```text
text_chunk
time_segment
frame
whole_resource
```

Для аудио и видео основная единица поиска — `time_segment`.

Для описания кадра — `frame`.

Для сравнения целых ресурсов — `whole_resource`.

## 5.4. Raw extraction atom

Это не core SearchUnit.

```python
@dataclass(frozen=True)
class TimedTextAtom:
    atom_id: str
    start_ms: int
    end_ms: int
    text: str
    token_count: int | None = None
    speaker: str | None = None
    confidence: float | None = None
    metadata: Mapping[str, JSONValue] = field(default_factory=dict)
```

Источники:

- Whisper segment;
- VTT cue;
- SRT cue;
- другой STT engine;
- внешняя БД;
- ручная транскрипция с таймингами.

Сырые atoms:

- хранятся как extraction artifact;
- могут использоваться для повторного chunking;
- не обязаны попадать в retrieval database;
- не должны напрямую определять стабильный public search contract.

## 5.5. TimedPassage

Основная media search unit:

```python
@dataclass(frozen=True)
class TimedPassage:
    passage_id: str
    start_ms: int
    end_ms: int
    text: str
    ordinal: int
    token_count: int
    source_atom_ids: tuple[str, ...]
    metadata: Mapping[str, JSONValue] = field(default_factory=dict)
```

Core projection:

```python
SearchUnitRecord(
    unit_id=passage.passage_id,
    resource_id=resource_id,
    representation_id=transcript_representation_id,
    unit_kind="time_segment",
    modality="text",
    text=passage.text,
    evidence_locator=Locator(
        "time_segment",
        {
            "start_ms": passage.start_ms,
            "end_ms": passage.end_ms,
            "track": "audio",
        },
    ),
    ordinal=passage.ordinal,
    token_count=passage.token_count,
    token_count_kind="exact",
)
```

## 5.6. FrameCaptionObservation

```python
@dataclass(frozen=True)
class FrameCaptionObservation:
    frame_id: str
    timestamp_ms: int
    caption: str
    token_count: int
    producer_fingerprint: str
    metadata: Mapping[str, JSONValue] = field(default_factory=dict)
```

Core projection:

```python
SearchUnitRecord(
    unit_id=...,
    resource_id=video_resource_id,
    representation_id=frame_caption_representation_id,
    unit_kind="frame",
    modality="text",
    text=caption,
    evidence_locator=Locator(
        "video_frame",
        {
            "timestamp_ms": observation.timestamp_ms,
            "frame_id": observation.frame_id,
        },
    ),
    ordinal=...,
)
```

Кадр и passage связываются по времени, а не через обязательную foreign-key relation.

# 6. Стандарт временных locator-ов

## 6.1. Единицы времени

Везде используются целые миллисекунды:

```text
start_ms
end_ms
timestamp_ms
duration_ms
```

Не использовать:

- float seconds;
- строки `HH:MM:SS` как storage contract;
- provider-specific time units.

## 6.2. Интервалы

Временные интервалы полуоткрытые:

```text
[start_ms, end_ms)
```

Инварианты:

```text
start_ms >= 0
end_ms > start_ms
```

## 6.3. Typed helpers

`mdrack_media` предоставляет:

```python
TimeSegmentLocator(start_ms, end_ms, track="audio")
VideoFrameLocator(timestamp_ms, frame_id)
WholeMediaLocator()
```

Они сериализуются в generic `mdrack_core.Locator`.

## 6.4. Почему связь только по timeline

Не создавать модель:

```text
passage.frames[]
```

в первой версии.

Причины:

- chunking policy может измениться;
- кадры извлекаются отдельно;
- один кадр может находиться около границы passages;
- frame search может сработать без речи;
- повторное chunking не должно требовать переиндексации frame artifacts;
- host может выбрать разные правила контекстуализации.

# 7. Политика группировки транскрипций

## 7.1. Зачем не индексировать Whisper atoms

Слишком мелкие units:

- дают слабый semantic context;
- увеличивают vectors и размер индекса;
- повышают число почти одинаковых результатов;
- ухудшают resource-level grouping;
- зависят от версии распознавателя;
- часто не являются законченными мыслями.

Whisper atoms сохраняют provenance и тайминги, но не являются основным retrieval granularity.

## 7.2. TokenCounter boundary

`mdrack_media` не импортирует tokenizer конкретной модели.

Интерфейс:

```python
class TokenCounter(Protocol):
    def count(self, text: str) -> int:
        ...
```

Возможные реализации вне pure media package:

- exact tokenizer выбранной embedding model;
- provider-side tokenizer;
- deterministic estimate.

`TimedTextAtom.token_count` может быть заполнен producer-ом заранее.

В persisted records обязательно сохраняется:

```text
token_count
token_count_kind = exact | estimated
```

## 7.3. Начальная chunking policy

Рекомендуемый default для первого evaluation:

```python
TimedChunkingPolicy(
    soft_min_tokens=180,
    target_tokens=320,
    soft_max_tokens=480,
    hard_max_tokens=800,

    soft_min_duration_ms=20_000,
    target_duration_ms=60_000,
    soft_max_duration_ms=90_000,
    hard_max_duration_ms=120_000,

    medium_pause_ms=700,
    strong_pause_ms=1_500,

    overlap_atoms=0,
)
```

Это не догма. Эти числа должны пройти A/B evaluation.

## 7.4. Boundary signals

Сильные границы:

- окончание предложения;
- смена speaker;
- pause >= `strong_pause_ms`;
- chapter boundary, если передан producer-ом;
- hard token/duration limit.

Средние границы:

- pause >= `medium_pause_ms`;
- перевод строки;
- soft max tokens;
- soft max duration.

Слабые границы:

- target tokens;
- target duration.

## 7.5. Детерминированный алгоритм

1. Проверить atoms.
2. Отсортировать или отклонить неупорядоченный input согласно strict mode.
3. Нормализовать только технические пробелы; не переписывать текст.
4. Начать candidate passage с первого atom.
5. На каждом atom рассчитывать:
   - текущие tokens;
   - duration;
   - gap до следующего atom;
   - sentence-end;
   - speaker change;
   - hard limit risk.
6. После `soft_min_*` собирать возможные boundary candidates.
7. Выбрать boundary с максимальным deterministic score в окне.
8. При достижении hard max закрыть passage независимо от punctuation.
9. `start_ms` взять у первого atom.
10. `end_ms` взять как max end последнего включённого atom.
11. Сформировать stable ID из:
    - resource identity;
    - chunker fingerprint;
    - ordinal;
    - first/last atom IDs;
    - start/end;
    - canonical text hash.
12. Продолжить со следующего atom.
13. Не добавлять overlap в v1.

## 7.6. Пример boundary score

```text
+100  hard limit reached
 +45  speaker change
 +40  sentence end
 +35  strong pause
 +20  soft max reached
 +15  medium pause
 +10  target tokens reached
 +10  target duration reached

 -distance_penalty from target tokens
 -distance_penalty from target duration
```

Точный score должен быть зафиксирован тестами и versioned fingerprint.

## 7.7. Overlap policy

Default:

```text
overlap = 0
```

Overlap добавляется только после доказанного улучшения retrieval.

Экспериментальные варианты:

```text
one source atom
one sentence
5 seconds
```

При overlap необходимо:

- исключить повторный RRF вклад одного logical evidence;
- измерить duplicate result rate;
- измерить рост DB;
- проверять точность locator-а;
- маркировать chunker fingerprint.

## 7.8. Работа с overlapping atoms

Некоторые STT/subtitle источники имеют пересекающиеся cues.

Правила:

- `end_ms > start_ms` обязательно;
- start times должны быть неубывающими;
- небольшое overlap не исправляется скрытно;
- passage range = min start / max end;
- точные source atom IDs сохраняются;
- повторяющиеся строки могут дедуплицироваться только отдельной versioned normalization policy;
- producer-specific repair происходит до grouper-а.

# 8. Изображения в текущем scope

## 8.1. Stable capability

```text
image
→ OCR/caption text вне core
→ один или несколько text representations
→ one whole_resource unit per representation
→ text embeddings
→ lexical / semantic / hybrid search
```

## 8.2. Что означает similarity изображений сейчас

Допустимые названия:

```text
textual image similarity
caption-based similarity
OCR-text similarity
```

Недопустимое обещание:

```text
visual similarity
```

если сравниваются только тексты.

## 8.3. Размер unit для изображения

Для OCR/caption текста default:

```text
one whole_resource unit
```

Предел:

```text
max_image_text_tokens = 8_000
```

Но token count должен быть:

- exact, если tokenizer доступен;
- estimated с safety margin, если нет.

Если текст превышает limit:

- не молча обрезать;
- либо отклонить;
- либо caller заранее создаёт несколько region/page text representations;
- core не выполняет скрытый chunking.

## 8.4. Existing visual code

Existing visual-vector abstractions:

- остаются experimental;
- не расширяются;
- не входят в media roadmap;
- не участвуют в default CLI;
- не используются в release claims;
- могут быть вынесены в optional integration позже.

# 10. Whole-resource текстовые representations

## 10.1. Зачем они нужны

Passage vectors находят место внутри ресурса.

Whole-resource vectors нужны для:

- похожих документов;
- похожих аудио по содержанию речи;
- похожих видео по транскрипции;
- похожих видео по описаниям кадров;
- resource-level recommendations.

## 10.2. Короткий resource

Если canonical text помещается в configured limit:

```text
canonical full text
→ one whole_resource unit
→ one text embedding
```

Начальный limit:

```text
whole_resource_text_max_tokens = 8_000
```

Не использовать максимальное context window модели автоматически. Limit является продуктовой policy.

## 10.3. Длинный resource

Три допустимых strategy, все выполняются вне core:

### Strategy A: passage vector centroid

```text
passage vectors
→ weighted mean
→ L2 normalization для cosine space
→ whole_resource vector
```

Fingerprint обязан включать:

```text
aggregation algorithm
version
weighting
normalization
source passage space fingerprint
```

### Strategy B: deterministic extract

```text
выбранные первые/ключевые passages
→ bounded canonical text
→ embedding
```

### Strategy C: caller-supplied summary

```text
external summarizer
→ summary text artifact
→ embedding
```

Core не знает, какой способ использован.

## 10.4. Рекомендуемый первый вариант

Для первого working slice:

- короткие ресурсы: embed canonical full text;
- длинные: normalized token-weighted mean passage vectors;
- в similarity result показывать лучшие matching passages как evidence;
- не считать centroid окончательной оценкой качества;
- измерять отдельно.

# 11. Поисковые режимы

## 11.1. Unit target

Возвращает конкретные:

- Markdown chunks;
- timed passages;
- frame captions;
- image descriptions.

Нужен для:

- precise evidence;
- seek;
- RAG context.

## 11.2. Resource target

Схлопывает units по `resource_id` внутри каждой branch, **перенумеровывает resource ranks**, затем выполняет fusion.

Нужен для:

- поиска файлов;
- рекомендации похожих ресурсов;
- UI lists;
- предотвращения доминирования длинных видео.

## 11.3. Text mode

FTS по text-bearing units.

Фильтры применяются до limit.

## 11.4. Semantic mode

Caller передаёт готовый text query vector.

Core проверяет:

- `space_id`;
- expected fingerprint;
- dimensions;
- finite values;
- zero-vector policy;
- adapter result contract.

## 11.5. Hybrid mode

Минимум:

```text
lexical branch
text semantic branch
```

Для видео можно иметь:

```text
combined lexical branch over transcript + frame captions
combined text semantic branch
```

Позже, если доказана необходимость различных весов по representations, добавить branch-local narrowing scope.

## 11.6. Branch-local scope

Текущий global `SearchScope` полезен как верхняя граница.

Для универсальной media-системы желательно добавить:

```python
LexicalBranch(
    ...,
    scope_override=SearchScope(
        representation_kinds=("transcript_text",),
    ),
)
```

и аналогично для `VectorBranch`.

Правило:

```text
effective scope = intersection(request.scope, branch.scope_override)
```

Branch override может только сужать глобальный scope, но не расширять его.

Это позволит отдельно взвешивать:

- transcript lexical;
- frame-caption lexical;
- transcript semantic;
- frame-caption semantic.

Добавлять контракт следует до core 1.0 либо отложить до измеренной необходимости. Решение фиксируется ADR.

## 11.7. Similarity result

Result должен содержать:

```text
resource_id
similarity_basis
space_id
score/rank
best evidence units
locators
degradations
```

Не использовать одно слово `similar` без basis.

# 29. Definition of Done для text-only media capability

Функция считается готовой, если:

1. External producer может передать timed atoms без знания DB.
2. Grouper создаёт deterministic timed passages.
3. Каждый passage имеет точные `start_ms/end_ms`.
4. Core индексирует prepared graph без provider calls.
5. SQLite сохраняет ресурс атомарно.
6. Text search возвращает passage.
7. Semantic search возвращает passage.
8. Hybrid search возвращает passage.
9. Resource search не зависит от количества units.
10. Frame caption возвращает timestamp.
11. Whole-resource similarity явно помечена как textual.
12. Source file и binary не хранятся в DB.
13. Logs не содержат content/path/query.
14. Offline E2E работает на Linux и Windows.
15. Installed package smoke работает вне source tree.
16. Curated corpus показывает зафиксированные metrics.
17. Index полностью перестраивается из external artifacts.
18. Future-host adapter proof не требует изменения core.

# 31. Финальная архитектурная формула

```text
Extractor/provider code
    создаёт тексты, atoms, captions и text vectors

mdrack-media
    превращает их в универсальные timed passages и resource graphs

mdrack-core
    валидирует, ищет, группирует, объединяет ranks и возвращает evidence

mdrack-sqlite
    хранит производный index

standalone MDRack или future host
    разрешает logical source locator в реальный файл и UI action
```

Ключевые ограничения:

```text
Whisper atom ≠ retrieval unit
Timed passage = основной unit речи
Frame caption = независимый unit на общей timeline
Whole-resource text vector = основа similarity целых файлов
Visual/audio vectors = не текущий roadmap
WVM = future compatibility target, не архитектурный владелец
```

Этот путь позволяет довести MDRack до самостоятельного зрелого проекта, проверить его на собственном корпусе, использовать лично и только затем подключить к WVM обычным adapter-ом — без переписывания retrieval-ядра и без превращения проекта в преждевременный мультимодальный космический корабль.
