# MDRack 1.2.0 — Unified Text Search Implementation Plan

> **Status:** ACTIVE PLAN for the MDRack 1.2.0 implementation wave.

**Goal:** Give users one text-first search entry over notes, audio transcripts, video transcripts, video-frame captions, and image text, plus provider-free “find similar” by logical resource ID.

**Architecture:** Reuse the active resource-core generation, existing `SearchScope`, SQLite catalog, retrieval/RRF owners, LM Studio query embeddings, and portable locators. Add one application-owned scope resolver and unified result mapping; preserve every v1.1 CLI/API path. No SQL migration or new ranker is planned, but an explicit rebuild is required for new whole-resource projections.

**Version:** `1.2.0` (new backward-compatible public capability over `1.1.0`).

**Research basis:** source-first audit at Git `ee5f9f1`, artifact SHA-256 `6db67b0a5f176282c28486df71918470472afcefe612c8ea46cc3fc5a57e0ae3`.

## User-visible contract

```text
mdrack search QUERY --scope all|notes|audio|video|frames|images
mdrack find-similar RESOURCE_ID --scope all|notes|audio|video|images
```

Additive embedded API:

```python
await engine.search_unified(query, scope="all", mode="hybrid", limit=20)
engine.find_similar_resource(resource_id, scope="all", limit=20)
```

- `all` searches text representations only; it does not compare pixels, audio waves, or visual vectors.
- `frames` is valid for query search and invalid for resource-level similarity.
- Similarity reuses a stored textual whole-resource vector and never calls an embedding provider.
- Missing textual vectors or an unavailable LM Studio query provider produce stable degraded results, not fabricated matches.
- Existing `search`, `search --catalog`, `image search`, `similar`, `resource similar`, and `resources similar` behavior remains compatible.

## Stage A — Scope and projection completeness

**Objective:** Make every promised resource kind searchable through one typed scope contract and eligible for deterministic textual similarity where supported.

**Likely files:**
- `src/mdrack/application/resources.py`
- `src/mdrack/application/compatibility.py`
- `src/mdrack/ingestion/images.py`
- focused unit/integration tests for those owners

**Work:**
1. Add a typed resolver for `all|notes|audio|video|frames|images` that compiles to existing `SearchScope` fields.
2. Add unified result/evidence DTOs using logical IDs and portable locators only.
3. Resolve a logical resource ID to exactly one compatible textual whole-resource unit for similarity.
4. Enable deterministic Markdown whole-resource projection through the existing compatibility seam.
5. Add one deterministic text-only aggregate for image caption/OCR text; never mix visual vectors into textual similarity.

**Acceptance:** alias matrix, deterministic IDs/fingerprints, provider-free similarity, visual-only image degradation, source immutability, and unchanged core/SQLite contracts.

## Non-goals and stop conditions

- No GUI, web server, MCP server, vector database, direct model loading, reranker, raw-media transcription, acoustic search, or pixel similarity.
- Do not modify `mdrack-core` ranking contracts, `mdrack-sqlite` schema/migrations, or existing v1.1 public result shapes unless a reproducible blocker proves the plan wrong.
- Stop for schema/public-contract incompatibility, source/data corruption risk, secret/live-cost boundary, or an implementation that requires multiple catalog locations in one request.
- Ordinary in-scope defects go through one bounded repair and fresh review; they are not owner gates.
