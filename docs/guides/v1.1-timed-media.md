# Timed transcript readers in MDRack 1.1

MDRack 1.1 can normalize four transcript formats into the existing
`mdrack_media.TranscriptArtifact` without running Whisper or touching a media
file. Reading and source identity remain the host application's responsibility.

## Basic use

```python
from mdrack.ingestion.transcripts import read_transcript
from mdrack_media import ProducerFingerprint, resource_id

resource = resource_id("my-host", "audio/episode-42")
producer = ProducerFingerprint.from_payload(
    {"engine": "whisper", "model": "large-v3", "config_version": 1}
)

result = read_transcript(
    transcript_bytes,
    resource_id=resource,
    producer_fingerprint=producer,
    language="ru",
)
artifact = result.artifact
```

Use the explicit `read_whisper_json`, `read_vtt`, `read_srt`, or
`read_timed_json` function when the format is already known. Autodetection is
content-based and supports the same four formats.

## Choosing strictness

Keep the default `strict=True` for indexing and reproducible builds. Invalid
JSON, malformed timestamps, empty text, reversed intervals, and overlap stop the
read with a stable `TranscriptReadError.code`.

Use `strict=False` only for an intentional salvage workflow:

```python
result = read_vtt(
    vtt_text,
    resource_id=resource,
    producer_fingerprint=producer,
    language="ru",
    strict=False,
)
for diagnostic in result.diagnostics:
    print(diagnostic.code, diagnostic.source_ordinal)
```

Diagnostics are safe to report because they contain no source text or raw
exception. The artifact itself intentionally contains normalized transcript
text and must be handled as user content.

## Identity and rebuilds

Keep `resource_id` stable for the same logical source. Change the producer
fingerprint whenever the effective transcription model or configuration
changes. MDRack generates representation and atom IDs from those identities and
canonical atom order.

`result.source_fingerprint` hashes the exact input bytes (or canonical JSON for
an already decoded mapping). Formatting-only JSON changes therefore retain
semantic atoms but change the source fingerprint. Reading the same input twice
produces the same artifact and diagnostics.

## Format notes

- Whisper JSON requires a `segments` array with second-based `start`, `end`, and
  `text`; `id`, `speaker`, and probability-valued `confidence` are optional.
- WebVTT accepts BOM, CRLF/LF, header metadata, cue IDs, cue settings, and
  multiline cue text. `NOTE`, `STYLE`, and `REGION` blocks are ignored.
- SRT accepts cue IDs and multiline text with comma millisecond timestamps.
- Generic JSON requires schema `mdrack.timed-transcript.v1` and integer
  millisecond atoms.

Readers collapse cue whitespace and remove subtitle cue markup. They do not
combine cues. Passage grouping and retrieval belong to later application
stages.

## Safety boundary

Pass source contents, not source paths. These readers never open files, infer
language from filenames, call network services, invoke FFmpeg/Whisper, emit raw
content to logs, or write back to the source. Read the file in the host layer,
apply host-specific size limits there, then pass bytes into the reader.
