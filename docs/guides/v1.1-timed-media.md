# Timed transcript ingestion and retrieval in MDRack 1.1

MDRack 1.1 can normalize four transcript formats into the existing
`mdrack_media.TranscriptArtifact` without running Whisper or touching a media
file. Reading and source identity remain the host application's responsibility.

## Basic use

```python
from mdrack.ingestion.transcripts import read_transcript
from mdrack_media import ProducerFingerprint, resource_id

resource = resource_id("my-host", "audio/episode-42")
producer = ProducerFingerprint.from_payload(
    {"engine": "whisper", "model": "large-v3", "config_version": 1}
)

result = read_transcript(
    transcript_bytes,
    resource_id=resource,
    producer_fingerprint=producer,
    language="ru",
)
artifact = result.artifact
```

Use the explicit `read_whisper_json`, `read_vtt`, `read_srt`, or
`read_timed_json` function when the format is already known. Autodetection is
content-based and supports the same four formats.

## Choosing strictness

Keep the default `strict=True` for indexing and reproducible builds. Invalid
JSON, malformed timestamps, empty text, reversed intervals, and overlap stop the
read with a stable `TranscriptReadError.code`.

Use `strict=False` only for an intentional salvage workflow:

```python
result = read_vtt(
    vtt_text,
    resource_id=resource,
    producer_fingerprint=producer,
    language="ru",
    strict=False,
)
for diagnostic in result.diagnostics:
    print(diagnostic.code, diagnostic.source_ordinal)
```

Diagnostics are safe to report because they contain no source text or raw
exception. The artifact itself intentionally contains normalized transcript
text and must be handled as user content.

## Identity and rebuilds

Keep `resource_id` stable for the same logical source. Change the producer
fingerprint whenever the effective transcription model or configuration
changes. MDRack generates representation and atom IDs from those identities and
canonical atom order.

`result.source_fingerprint` hashes the exact input bytes (or canonical JSON for
an already decoded mapping). Formatting-only JSON changes therefore retain
semantic atoms but change the source fingerprint. Reading the same input twice
produces the same artifact and diagnostics.

## Format notes

- Whisper JSON requires a `segments` array with second-based `start`, `end`, and
  `text`; `id`, `speaker`, and probability-valued `confidence` are optional.
- WebVTT accepts BOM, CRLF/LF, header metadata, cue IDs, cue settings, and
  multiline cue text. `NOTE`, `STYLE`, and `REGION` blocks are ignored.
- SRT accepts cue IDs and multiline text with comma millisecond timestamps.
- Generic JSON requires schema `mdrack.timed-transcript.v1` and integer
  millisecond atoms.

Readers collapse cue whitespace and remove subtitle cue markup. They do not
combine cues. Passage grouping and retrieval belong to later application
stages.

## Ingest a complete transcript graph

The application service composes the readers with the existing deterministic
`mdrack-media` builders and the clean catalog. A caller supplies a canonical
`resource_<sha256>` ID (create one with `mdrack_media.resource_id`), a portable
core `Locator`, and an already read artifact. Provider calls finish before the
single complete `replace_resource` transaction:

```python
from mdrack.application.transcript_ingestion import TranscriptIngestionService
from mdrack_core import Locator

service = TranscriptIngestionService(
    catalog,
    embedding_provider=provider,
    embedding_fingerprint=embedding_profile_fingerprint,
)
result = await service.ingest(
    artifact,
    resource_kind="audio",
    media_type="audio/wav",
    source_namespace="recordings",
    source_locator=Locator("external_record", {"source_ref": "lecture-42"}),
)
```

Pass `embeddings=False` to create a lexical-only resource. Repeating ingestion
with ready vectors replaces the full resource graph atomically. Repeating it
without vectors removes the prior vector rows; this is complete replacement,
not an incremental patch.

The built-in `DeterministicWhitespaceCounter` is an estimate: its representation
and unit records persist `token_count_kind="estimated"`, and that provenance is
part of the grouping fingerprint and logical identities. A host that injects a
provider tokenizer must also pass `token_count_kind="exact"` (or explicitly
`"estimated"` for another approximation); MDRack never infers exactness from a
custom counter.

`MDRackEngine.ingest_transcript()` and `MDRackEngine.search_transcripts()` expose
the same orchestration for embedded hosts. Text, semantic, and hybrid results
share `TimedSearchResult`; each evidence item includes logical unit and
representation IDs, `start_ms`, `end_ms`, `track`, and `timestamp_unit="ms"`.
If embeddings or a compatible vector space are unavailable, semantic search
returns a safe empty degradation and hybrid search falls back to lexical hits.
Vector-space discovery uses the provider-neutral catalog
`resolve_embedding_space(fingerprint=..., dimensions=...)` capability. Application
code does not inspect SQLite connections or tables. Each timed lexical and
semantic branch requests `max(100, result_limit * 10)` candidates before core
resource grouping and fusion, so passage-heavy resources cannot consume the
entire final-result budget prematurely.

## CLI offline workflow

I1 exposes the clean-catalog vertical slice explicitly:

```bash
mdrack ingest transcript lecture.whisper.json \
  --resource-id resource_<canonical-sha256> \
  --kind audio \
  --media-type audio/wav \
  --namespace recordings \
  --source-ref lecture-42 \
  --no-embeddings \
  --catalog ./catalog.sqlite3

mdrack search "transaction boundary" \
  --mode text \
  --kind audio \
  --catalog ./catalog.sqlite3
```

Use `--provider fake` for deterministic offline semantic/hybrid verification or
the configured LM Studio provider for an explicitly authorized local-provider
run. `--format` accepts `auto`, `whisper-json`, `vtt`, `srt`, and
`timed-json-v1`. The command reads a bounded transcript file but stores only the
prepared text graph; it neither opens a media binary nor writes to the source.

## Safety boundary

Readers never open files, infer language from filenames, call network services,
invoke FFmpeg/Whisper, emit raw content to logs, or write back to the source.
Read the file in the host layer, apply host-specific size limits there, then pass
bytes into the reader. The CLI is that host layer for its explicit ingestion
command and enforces the application manifest byte limit before parsing.
