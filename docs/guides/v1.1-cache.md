# Using the MDRack 1.1 artifact cache

The cache stores reusable derived artifacts outside the retrieval catalog. See the [artifact cache contract](../contracts/v1.1-artifact-cache.md) for identity, immutability, privacy, and recovery guarantees.

## Configure a cache

The default cache is enabled at `.mdrack/artifacts`, resolved relative to the selected project root:

```toml
[cache]
enabled = true
directory = ".mdrack/artifacts"
max_entry_bytes = 104857600
```

Environment overrides follow the normal configuration convention:

```text
MDRACK_CACHE_ENABLED=false
MDRACK_CACHE_DIRECTORY=/var/cache/mdrack/artifacts
MDRACK_CACHE_MAX_ENTRY_BYTES=52428800
```

Use a dedicated cache directory, not the project root, store root, database parent, or a directory shared with foreign content. Cache payloads can contain transcript text, captions, or vectors, so filesystem access to the directory must be treated as access to source-derived private data. The default `.mdrack/artifacts` is a dedicated child of the store and is safe to purge independently.

## Inspect without exposing content

```bash
mdrack --root /project cache status
mdrack --root /project cache verify
```

Both commands return JSON envelopes with counts only. `verify` reads and hashes every entry. A non-zero exit means at least one entry is corrupt or verification could not complete; it does not delete the entry.

Normal application lookup recovers safely: a corrupt entry is discarded, the lookup behaves as a miss, and the owning producer can rebuild it. Source files and retrieval catalogs are not read or modified by cache recovery.

## Integrate an optional text provider

Durable caching is opt-in. Construct `ArtifactCache`, inject it into the provider, and pass a `ProviderCacheContext` for each request. The context must contain opaque SHA-256 fingerprints for source content, producer, actual prompt, caller configuration, and preprocessing, plus a safe artifact kind.

A complete matching identity returns the immutable cached provider envelope and does not call the transport. Source, producer, model, prompt, configuration, preprocessing, or runtime-option drift produces a miss. Calls without `cache_context` retain the bounded process-local cache and do not write durable artifacts.

## Purge is separate and destructive

The guarded command is:

```bash
mdrack --root /project cache purge --confirm
```

Without `--confirm`, MDRack returns `CACHE_CONFIRMATION_REQUIRED` and changes nothing. A confirmed purge requires the exact MDRack marker and an exclusively cache-owned layout. It rejects project/store/catalog roots, missing or forged markers, foreign content, and unsafe symlinks. It removes validated cache shards and retains the dedicated root plus marker. There is no automatic purge during rebuild, rollback, release, or normal provider work.

Do not run purge as a routine repair for one corrupt entry; normal lookup already removes only that failed identity. Back up or regenerate artifacts according to the owning producer's policy when they are expensive or impossible to reproduce.

## Evidence limits

The automated cache suite is offline and uses temporary files plus injected provider transports. Confirmed purge is executed only against disposable fixtures. The suite does not contact LM Studio, OpenRouter, paid endpoints, private corpora, or Windows, and it does not delete a real project cache.
